# 백업/복구 리허설 런북 (Restore Drill Runbook)

본 런북은 와이앤아처 통합 Works 플랫폼의 데이터 복구 절차와 분기 1회 복구 리허설 수행 방법을 정의합니다. 목표 지표는 **RPO 24시간 / RTO 4시간**이며, 상세 보존 정책은 [5_backup_retention_privacy.md](../../docs/docs_dev/5_backup_retention_privacy.md)를 따릅니다.

---

## 1. 백업 구성 요약

* **데이터베이스(Supabase/PostgreSQL)**: 일 1회 자동 백업(PITR 옵션 활성 시 시점 복구 가능). 백업 주기가 RPO 24h를 만족합니다.
* **오브젝트 스토리지(S3)**: 버전 관리(Versioning) 활성 + 교차 리전 복제(CRR) 권장. 정적 산출물은 CI 재빌드로 재생성 가능.
* **마이그레이션(SSOT)**: `supabase/migrations/`가 스키마 원장이며, Git이 곧 스키마 백업입니다.

---

## 2. 복구 절차 (RTO 4h 이내)

1. **영향 범위 확정**: 손상 대상(DB/스토리지/배포)과 목표 복구 시점(RPO)을 선정합니다.
2. **DB 복구**: Supabase 콘솔에서 대상 시점 백업을 선택해 신규 인스턴스로 복원하거나, PITR로 특정 시점을 복구합니다.
3. **스키마 정합성 확인**: `supabase db diff`로 복원본과 `migrations/` 최신 상태의 드리프트를 점검하고, 누락분은 `supabase db push`로 재적용합니다.
4. **스토리지 복구**: S3 버전 관리에서 삭제/손상 객체를 이전 버전으로 복원하거나 CRR 리전에서 동기화합니다.
5. **프론트엔드 재배포**: `Deploy` 워크플로를 수동 실행(`workflow_dispatch`)해 최신 SPA를 재배포하고 CloudFront를 무효화합니다.
6. **RLS/권한 스모크 테스트**: pgTAP 회귀 테스트(테스트 계정 10종 + 보안 케이스 8종)를 실행해 접근 통제가 정상인지 확인합니다.

---

## 3. 분기 복구 리허설 체크리스트

> 분기 1회, 운영과 분리된 스테이징 프로젝트에서 수행하고 결과를 기록합니다.

- [ ] 최신 자동 백업으로 스테이징 DB 복원 성공(소요 시간 기록)
- [ ] `supabase db diff` 드리프트 0 확인
- [ ] S3 객체 버전 복원 샘플 검증(임의 5개 파일)
- [ ] SPA 재배포 + CloudFront 무효화 후 딥링크(예: `/mna/:id`) 정상 로딩
- [ ] pgTAP RLS 회귀 테스트 전건 통과
- [ ] **측정 RTO ≤ 4h / RPO ≤ 24h** 충족 여부 판정 및 회고
- [ ] 실패/지연 항목 이슈 등록 및 개선 액션 지정
