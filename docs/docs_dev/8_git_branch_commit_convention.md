# [8] Git 브랜치 전략 및 커밋 컨벤션 정의서

본 문서는 와이앤아처 통합 Works 플랫폼 모노레포의 **형상 관리 표준**을 정의합니다. 브랜치 운영 전략, 커밋 메시지 규칙, 풀 리퀘스트(PR) 절차를 통일하여 변경 이력의 추적성과 협업 안정성을 확보합니다.

---

## 1. 목적 및 범위

* **목적**: 다수의 작업 주체(사람 및 AI 에이전트)가 동일 저장소에 기여할 때 발생하는 이력 혼선을 방지하고, 릴리스 가능한 `main` 브랜치를 항상 안정 상태로 유지합니다.
* **범위**: `ynarcher_works` 모노레포의 모든 워크스페이스(`apps/*`, `packages/*`, `docs/*`, `supabase/*`)에 공통 적용합니다.
* **원격**: `https://github.com/ynarcher-dev/ynarcher_works.git`

---

## 2. 브랜치 전략 (Trunk-based)

짧게 유지되는 기능 브랜치를 `main` 트렁크에 병합하는 **트렁크 기반 개발**을 채택합니다.

| 브랜치 | 역할 | 규칙 |
| :--- | :--- | :--- |
| **`main`** | 릴리스 가능한 단일 통합 트렁크 | 보호 브랜치. 직접 푸시 금지, PR 병합으로만 갱신 |
| **기능 브랜치** | 단위 작업(기능/수정/문서) | `main`에서 분기하여 작업 후 PR로 병합, 병합 후 삭제 |

### 2.1 기능 브랜치 네이밍

* **형식**: `<type>/<scope>-<간단설명>` (영문 소문자/하이픈). 예: `feat/ac-evaluation-engine`, `fix/office-calendar-timezone`, `docs/dev-git-convention`
* AI 에이전트 자동화 세션은 지정된 작업 브랜치 이름(예: `claude/<설명>`)을 그대로 사용할 수 있습니다.

> [!NOTE]
> 기능 브랜치는 **짧게 유지**합니다. 장수 브랜치는 병합 충돌과 이력 드리프트의 원인이 되므로, 작업 단위를 잘게 쪼개어 자주 병합합니다.

### 2.2 병합 방식

* PR은 **Squash Merge**를 기본으로 하여 `main`의 이력을 작업 단위별 단일 커밋으로 선형 유지합니다.
* 병합된 PR은 완료된 것으로 간주하며, 후속 작업은 새 브랜치에서 진행합니다. (병합된 브랜치에 새 커밋을 쌓지 않습니다.)

## 3. 커밋 컨벤션

### 3.1 형식

```
<type>(<scope>): <제목>

<본문(선택)>
```

* **제목과 본문은 한국어**로 작성하며, 제목은 명령형 요약(마침표 없음)으로 50자 내외를 권장합니다.
* 본문은 "무엇을·왜"를 설명하고, 목록은 `-` 불릿을 사용합니다.

### 3.2 `type` 목록

| type | 용도 |
| :--- | :--- |
| **`feat`** | 새로운 기능 추가 |
| **`fix`** | 버그 수정 |
| **`docs`** | 문서 추가/수정 |
| **`refactor`** | 동작 변화 없는 코드 구조 개선 |
| **`chore`** | 빌드/설정/의존성 등 부수 작업 |
| **`test`** | 테스트 추가/수정 |

### 3.3 `scope` 목록

* **워크스페이스/영역**: `office`, `startup`, `networks`, `ac`, `fund`, `mna`, `admin`, `project`, `management`, `guest`
* **공통 영역**: `ui`, `master-data`, `db`, `auth`, `deps`
* **문서**: 문서 수정은 `docs(<폴더>)` 형태로 폴더를 스코프에 표기합니다. 예: `docs(docs_dev)`, `docs(master)`

### 3.4 예시

```
feat(ac): 공통 평가 엔진 가중치 집계 로직 추가

- evaluation_targets 다형 참조 기반 라운드 집계 함수 구현
- 심사위원 미제출 항목은 집계 모수에서 제외
```

```
docs(docs_dev): Git 브랜치 전략 및 커밋 컨벤션 정의서 신설
```

## 4. 풀 리퀘스트(PR) 절차

1. 기능 브랜치에서 작업을 완료하고 로컬 검증(`pnpm lint`, `pnpm typecheck`, `pnpm build`)을 통과시킵니다.
2. `main` 대상으로 PR을 생성하고, 제목은 커밋 컨벤션과 동일한 형식을 따릅니다.
3. 본문에는 변경 요약·검증 방법·영향 범위를 기재합니다.
4. 리뷰 승인 후 **Squash Merge**로 병합하고 브랜치를 삭제합니다.

> [!NOTE]
> DB 스키마 변경 PR은 반드시 `supabase/migrations/`의 순차 마이그레이션 파일을 포함해야 하며, 운영 DB 직접 수정은 금지합니다. (docs_dev/7_database_design_guidelines.md)

---

## 5. 커밋 전 로컬 검증 체크리스트

* [ ] `pnpm lint` — ESLint 통과 (의존성 경계 규칙 포함)
* [ ] `pnpm typecheck` — 타입 오류 없음
* [ ] `pnpm build` — 전 워크스페이스 빌드 성공
* [ ] 변경과 무관한 파일(빌드 산출물, `.env.local`)이 포함되지 않았는지 확인
