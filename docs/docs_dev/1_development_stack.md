# [1] 와이앤아처 통합 Works 플랫폼 기술 스택 및 핵심 설계 의견서

본 문서는 **와이앤아처 통합 Works 플랫폼**의 기술 스택 선정 과정에서 사용자가 확정한 기술(React, TypeScript, AWS S3, CloudFront)을 바탕으로, 인증 모델의 이원화, 모노레포 내 의존성 경계 규칙, 공통 권장 라이브러리 및 핵심 개발 수칙을 정리한 의견서입니다.

*(스타일링 및 디자인 사양은 기존 디자인 가이드 폴더의 정책을 따르므로 본 의견서에서는 제외하였습니다.)*

---

## 📌 1. 사용자 확정 스택 분석

* **화면 프레임워크**: **React (Vite SPA)**
  * **의견**: Vite 기반의 React SPA는 빌드 및 개발 서버 구동 속도가 매우 빠르며, 사내 PMS 시스템처럼 내부 임직원이 장시간 페이지에 머무르는 업무형 대시보드 구조에 적합합니다. SEO(검색엔진 최적화)가 중요하지 않은 시스템이므로 SPA 구조가 완벽히 대응합니다.
* **개발 언어**: **TypeScript**
  * **의견**: 스타트업, 심사역, 펀드, 프로그램 등 복잡한 데이터 모델 간의 외래키 연동이 잦은 PMS 특성상, 정적 타입을 지원하는 TypeScript 도입은 AI 협업(바이브 코딩) 및 런타임 버그 방지를 위해 필수적입니다.
* **파일 저장소**: **AWS S3**
  * **의견**: 대용량 첨부파일, 계약서, 심사 보고서 등을 저렴하고 안전하게 보관할 수 있습니다. 단, 클라이언트에서 S3에 직접 업로드하는 경우 보안을 위해 **Presigned URL**을 발급받아 업로드하는 방식을 권장합니다.
* **배포 환경**: **AWS S3 + CloudFront**
  * **의견**: 서버 운영 및 유지 관리 비용을 최소화(정적 호스팅 비용 수준)할 수 있는 최적의 조합입니다. HTTPS 통신과 글로벌 캐싱(CDN)을 지원하여 빠른 응답 속도를 확보할 수 있습니다.

## 🔑 2. 인증 및 세션 모델 설계 제안 (이원화 인증 구조)

### 2.1 이원화 인증 구조의 기술적 리스크
* **인증 관리의 복잡도 증가**: 플랫폼 내에서 임직원 대상의 일반적인 Supabase Auth(이메일/비밀번호) 표준 JWT 세션을 사용하는 영역과 외부 참가자(스타트업/전문가) 대상의 OTP 검증 및 Edge Function 서명 기반 **커스텀 JWT**를 사용하는 영역이 공존하는 구조입니다.
* 프론트엔드의 Supabase Client나 API 요청 레이어에서 두 종류의 세션 관리(세션 리프레시 주기, 로컬스토리지 토큰 주입 등)가 체계적으로 분리되지 않으면 토큰 오염 및 RLS 정책 우회 오류가 발생하기 쉽습니다.

### 2.2 구체적인 구현 및 보완 제안
1. **인증 처리 레이어 추상화 (Unified Auth Interface)**:
   * 프론트엔드 클라이언트가 Supabase Auth SDK에 직접 의존하지 않도록, `authStore`나 공통 `AuthService` 인터페이스를 설계합니다.
   * 이를 통해 일반 로그인(ID/PW 표준 JWT)과 간편 로그인(OTP 커스텀 JWT)이 각각 다른 토큰 보관 및 갱신(Refresh) 생명 주기를 갖더라도, 하위 비즈니스 컴포넌트는 단일화된 세션 상태(`currentUser`, `userRole`)만 관찰하여 개발 복잡도를 낮출 수 있습니다.
2. **RLS 검증 헬퍼의 단일화 및 모듈화**:
   * PostgreSQL/Supabase 내 RLS 정책 수립 시 `auth.uid()`를 직접 사용하는 테이블과 커스텀 JWT 클레임(`participant_id`)을 검사해야 하는 테이블이 섞이게 됩니다.
   * RLS 권한 판단을 일원화하기 위해 **`current_app_user_id()`** 및 **`current_app_role()`** 헬퍼 함수를 별도 DB 공통 스키마에 패키징하여, 모든 테이블의 RLS 정책이 이 헬퍼 함수만 경유하도록 표준화해야 합니다.
3. **실시간 세션 무효화 장치 (Session Versioning)**:
   * 커스텀 JWT는 Supabase 세션 관리의 통제를 받지 않으므로 강제 로그아웃 처리가 어렵습니다.
   * 커스텀 JWT 생성 시 `public.users.session_version` 값을 페이로드에 포함하여 발행하고, DB RLS 헬퍼 함수 내에서 쿼리 실행 시마다 해당 토큰 내 버전과 DB 상의 실시간 `session_version` 값을 대조하여 일치하지 않으면 즉각 권한을 차단하도록 설계해야 안전합니다.

## 📦 3. 모노레포 의존성 경계 및 UI 컴포넌트 격리 제안

### 3.1 의존성 경계 훼손 리스크
* 개발 속도를 높이기 위해 `packages/ui` 내부의 개별 UI 컴포넌트에 데이터 조회 로직(예: 부서 선택용 Picker 내부에 Supabase Client 또는 react-query 호출을 내장)을 넣기 시작하면 공통 패키지의 독립성과 재사용성이 훼손됩니다.
* 이 경우 UI 컴포넌트가 특정 DB 스키마나 API 도메인에 종속되어, 다른 앱(`apps/guest`)이나 이기종 서비스에서 해당 패키지를 재사용하기가 불가능해집니다.

### 3.2 구체적인 구현 및 보완 제안
1. **컴포넌트 역할의 명확한 이원화**:
   * **`packages/ui` (Pure UI Layer)**: 데이터 결합이 전혀 없는 순수 디자인 토큰 기반 Atom 컴포넌트(Button, Input, Layout Shell 등)만 격리합니다.
   * **`packages/master-data` 또는 `apps/*/components` (Connected UI Layer)**: 비즈니스 데이터(Supabase Client, API Fetching)가 엮인 Picker, Auto-Complete Selector 등은 UI 패키지가 아닌 별도의 마스터 데이터 패키지나 로컬 앱 영역에 배치하여 의존성 방향(`apps → packages/ui`)을 깨끗하게 유지합니다.
2. **ESLint를 통한 의존성 주입 방지 규칙 강제**:
   * 개발자들의 정기적인 소스코드 파일 모니터링에 의존하기보다, ESLint 및 Turborepo의 의존성 검증 규칙(예: `eslint-plugin-import`의 `no-restricted-imports`)을 활용하여 `packages/ui` 내부에서 `@supabase/supabase-js` 또는 `@tanstack/react-query` 등을 참조할 시 빌드 단계에서 에러를 뿜도록 설정합니다.
3. **Compound Component / Slot 패턴 채택**:
   * Prop Drilling을 방지하면서 UI 컴포넌트를 깨끗하게 유지하기 위해, 복잡한 비즈니스 폼 UI 조립 시 `children`이나 `slot` 구조를 적극 채택합니다.
   * API 호출과 상태 관리는 상위 컨테이너(App 레이어의 Hook)가 담당하고, `packages/ui`는 단지 화면 골격과 스타일 레이아웃만 잡아주는 형태로 데이터를 주입받아 그리도록 설계합니다.

## ⚙️ 4. 공통 추천 라이브러리 스택 제안

와이앤아처 통합 Works 플랫폼에 최적화된 고생산성 패키지를 통합 채택할 것을 권장합니다.

1. **데이터 페칭**: **`@tanstack/react-query` (TanStack Query)**
   * **이유**: 조회 데이터 캐싱, 비동기 상태 관리 및 뮤테이션 후 실시간 목록 갱신에 최적입니다.
2. **전역 상태 관리**: **`Zustand`**
   * **이유**: 보일러플레이트가 적어 PMS 개발 속도를 극대화하며 가벼운 상태(사용자 정보, 토글 등) 관리에 용이합니다.
3. **폼 처리 및 유효성 검증**: **`React Hook Form` + `Zod`**
   * **이유**: 많은 입력 폼을 렌더링 지연 없이 비제어 상태로 처리하고 스키마 기반 유효성 검증을 함께 구현하기에 가장 이상적입니다.
4. **날짜 연산**: **`Day.js`**
   * **이유**: 프로그램 일정, 멘토링 매칭 시 타임존 변환 및 포맷팅에 매우 가벼운 경량 유틸입니다.
5. **라우팅**: **`react-router-dom` (v6)**
   * **이유**: SPA 중첩 라우팅 구조와 역할 기반 가드 구성에 용이합니다.
6. **스타일링**: **`Tailwind CSS`**
   * **이유**: 디자인 가이드 폴더(`docs_design`)의 토큰 및 컴포넌트 규격이 Tailwind 유틸리티 클래스와 `tailwind.config.js` 확장을 전제로 작성되어 있으므로, Tailwind CSS를 공식 스타일링 스택으로 확정합니다. 색상/타이포/모션/z-index 토큰은 디자인 문서의 수치를 config로 이관하여 사용합니다.

## 📏 5. 소스코드 유지보수 및 공통 개발 수칙

장기 유지보수 및 바이브 코딩(AI 에이전트 협업)의 안정성을 높이기 위해 엄격히 지켜야 할 개발 규약입니다.

### 5.1 파일당 코드 길이 제한 규칙 (Max Lines per File)
* **어떠한 소스코드 파일도 500줄을 넘길 수 없습니다.**
* 코드가 400줄에 임박하면 즉시 Hook, Component, Actions 등으로 리팩토링 및 분할을 검토합니다.
  * 단일 React 컴포넌트: **250줄 이하**
  * 단일 Custom Hook: **150줄 이하**
  * 단일 일반 함수: **50줄 이하**

### 5.2 데이터베이스(DB) 직접 수정 절대 금지
* 실제 운영 중인 DB 콘솔에 수동 접속하여 데이터를 수정 또는 삭제하는 행위를 금지합니다.
* 모든 스키마 구조 변경은 `supabase/migrations/`에 위치하는 순차 번호가 기입된 정식 마이그레이션 파일(`YYYYMMDDHHMMSS_*.sql` 등)을 통해서만 적용되어야 합니다.

### 5.3 RLS(Row Level Security) 필수 적용
* Supabase 내 모든 테이블은 Row Level Security를 필수로 활성화합니다.
* 사용자는 사전에 약속된 DB 헬퍼 함수(`current_app_user_id()`) 검증을 거친 권한(부서, 역할 등)에 맞춰 격리된 데이터에만 조회 및 쓰기가 가능해야 합니다.

### 5.4 개인정보 마스킹 처리 의무
* 사용자 목록, 이메일, 전화번호 등 개인 식별 정보는 목록 화면 출력 시 기본 마스킹 처리합니다.
  * **전화번호**: 중간 자리를 가림 (`010-****-5678`)
  * **이메일**: 아이디 앞 글자만 남김 (`h***@example.com`)
* 상세 권한이 확보된 담당자만 감사 로그(Audit Log)를 남긴 뒤 복호화된 원본 상세 정보를 볼 수 있습니다.

### 5.5 비동기 예외 처리 및 환경변수 완벽 격리
* 모든 외부 API 호출 및 DB 쿼리 연동은 반드시 `try-catch` 구문으로 예외를 처리하고 화면에 에러 안내(Alert 등)를 띄워 상태 먹통을 방지합니다.
* Supabase URL, API Key 및 S3 자격증명 등 중요 설정 정보는 하드코딩하지 않고 `.env.local` 환경 변수를 통해서만 주입받도록 구성합니다.

### 5.6 UI 컴포넌트 설계 및 재사용 원칙 (컴포넌트 인라인 생성 금지)
* **초기 컴포넌트 설계 선행**: 시스템 혹은 특정 화면 기능을 설계할 때, 코딩 시작 전 화면을 구성할 컴포넌트 단위를 사전에 먼저 설계하고 개발을 시작해야 합니다.
* **컴포넌트 인라인 생성 절대 금지**: 단일 소스코드 파일 내에 일회성 용도로 컴포넌트를 인라인(예: 동일 파일 내부의 `const LocalComponent = ...`)으로 선언하여 사용하는 행위를 엄격히 금지합니다.
* **컴포넌트 재사용 의무화**: 모든 컴포넌트는 재사용이 가능하도록 독립된 파일로 분리하여 작성하고, 반복되는 UI 패턴은 공통 컴포넌트로 정교하게 추상화하여 중복을 최소화합니다.

## 🛠️ 6. S3 + CloudFront 환경에서의 테크니컬 고려사항

### 6.1 CloudFront 404 Error (SPA Routing Fallback) 설정
* **문제 상황**: React Router를 사용하는 SPA는 `/projects/1`과 같은 경로로 직접 브라우저에 주소를 입력하고 새로고침할 경우, S3 버킷 내에 물리적으로 해당 폴더/파일이 없으므로 CloudFront가 **403 Forbidden** 또는 **404 Not Found**를 반환합니다.
* **해결책**:
  * CloudFront의 **Error Pages(오류 페이지)** 설정에서 `403` 및 `404` 에러에 대한 사용자 지정 에러 응답을 생성합니다.
  * 응답 페이지 경로를 **`/index.html`**로 지정하고 상태 코드를 **`200 OK`**로 강제 반환하도록 구성해야 합니다.

### 6.2 S3 첨부파일 보안 업로드 (Presigned URL)
* **문제 상황**: 프론트엔드 클라이언트가 S3 버킷에 직접 접근하도록 AWS Access Key를 코드에 내장하면 보안에 매우 취약합니다.
* **해결책**:
  * Supabase **Edge Functions (Deno)** 또는 내부 API 서버에 S3 업로드 권한을 부여합니다.
  * 클라이언트가 파일 업로드를 요청하면, 서버가 짧은 만료시간(예: 5분)을 가진 **S3 Presigned URL**을 생성하여 클라이언트에 전달합니다.
  * 클라이언트는 이 1회용 URL을 사용하여 안전하게 S3에 파일을 직접 업로드합니다.
