# [6] API 계약 및 서버 액션 보안 기준서

## 1. 문서 목적 및 개요

본 문서는 와이앤아처 통합 Works 플랫폼의 클라이언트와 백엔드 간 통신(Supabase Edge Functions, PostgreSQL RPC, REST API 등) 시 준수해야 하는 API 공통 명세와 보안 규격을 정의한다. 

모든 백엔드 개발자는 이 기준을 엄격하게 구현에 적용하여 일관된 에러 핸들링과 보안 경계를 확보해야 한다.

> [!NOTE]
> 본 플랫폼의 프론트엔드는 Vite 기반 SPA로 자체 서버 런타임을 갖지 않는다(`1_development_stack.md` 참조). 따라서 본 문서에서 말하는 **'서버 액션'은 Supabase Edge Function(Deno) 및 DB RPC 함수로 구현되는 서버 사이드 처리 단위**를 지칭한다.

---

## 2. API 설계 공통 보안 원칙

1. **클라이언트(UI) 무신뢰**: API는 클라이언트에서 전달하는 권한 상태값이나 사용자 역할을 전적으로 신뢰하지 않는다. 모든 권한 검증은 서버 환경 내에서 현재 세션 토큰을 바탕으로 직접 재검증한다.
2. **서버 사이드 권한 체크**: 데이터베이스 RLS가 최종 보안을 담당하더라도, 불필요한 DB 트래픽 방지와 명확한 에러 응답을 위해 API 미들웨어 및 서버 액션 가드 단에서 세션과 권한을 1차 확인한다.
3. **통일된 에러 규격**: 권한 오류나 검증 실패 발생 시, 시스템 상세 스택이 외부로 누출되지 않도록 표준화된 공통 에러 코드를 사용하여 응답한다.
4. **민감 액션의 감사 추적**: 비즈니스에 영향이 크거나 개인정보를 다루는 API 호출 시 감사 로그(`Audit Log`)를 적재하는 트리거를 연동한다.

---

## 3. 공통 응답 형식 (Response Format)

모든 API와 서버 액션은 다음과 같은 일관된 구조의 JSON 응답을 반환한다.

### 3.1 성공 시 (Success Response)
```json
{
  "ok": true,
  "data": {
    "id": "7ca6473e-327c-4e7a-9769-e77a28e83344",
    "name": "와이앤아처 보육 프로그램"
  }
}
```

### 3.2 실패 시 (Failure Response)
```json
{
  "ok": false,
  "error": {
    "code": "permission_denied",
    "message": "해당 메뉴 및 리소스에 대한 접근 권한이 없습니다."
  }
}
```

## 4. 공통 에러 코드 명세

API 처리 중 예외 발생 시 반환할 표준 에러 코드와 정의는 다음과 같다.

| 에러 코드 (`code`) | HTTP 매핑 | 발생 사유 및 적용 정의 |
| :--- | :---: | :--- |
| **`unauthenticated`** | 401 | 로그인 세션이 없거나 세션 토큰이 만료되어 인증을 통과하지 못한 경우 |
| **`permission_denied`** | 403 | 로그인은 되었으나 요청한 워크스페이스 권한 및 데이터 Scope 범위를 벗어난 경우 |
| **`validation_error`** | 400 | 클라이언트가 전송한 데이터의 타입이 맞지 않거나 필수 값이 누락된 경우 (Zod 유효성 검사 실패) |
| **`not_found`** | 404 | 요청한 ID에 해당하는 데이터 레코드나 리소스가 데이터베이스에 존재하지 않는 경우 |
| **`conflict`** | 409 | 이미 중복 등록된 데이터가 있거나 결재 진행 중 상태 등 레코드 상태 불일치 오류 |
| **`expired`** | 410 | 임시 게스트 공유 링크의 만료 일자가 경과하였거나 OTP 제한 기한이 만료된 경우 |
| **`rate_limited`** | 429 | 동일 IP 또는 사용자 계정으로부터 비정상적으로 잦은 API 요청이 인입된 경우 |
| **`internal_error`** | 500 | 데이터베이스 장애 등 서버 내부의 통제할 수 없는 런타임 오류가 발생한 경우 |

## 5. API 명세서 작성 가이드라인 (보안 메타 데이터 필수화)

개발 단계에서 신규 API 및 서버 액션을 개설하고 문서를 작성할 때, 명세 정보에 다음 보안 항목을 메타 정보로 반드시 포함해야 한다:
* **필요 워크스페이스 권한**: `ac`, `fund`, `mna` 등 요구되는 도메인 명시 및 읽기/쓰기 조건 기록
* **필요 데이터 Scope**: `global`, `department`, `program`, `company`, `self` 중 해당 API가 요구하는 접근 반경 지정
* **외부 사용자 허용 여부**: 스타트업/전문가 계정이 외부 게스트 UI를 통해 호출 가능한 API인지 여부 (`true`/`false`)
* **Audit Log 필요 여부**: 호출 시 감사 로그 적재 테이블에 이력을 강제 적재해야 하는 대상인지 여부
* **다운로드/Export 여부**: 대량 반출 및 파일 다운로드가 포함되어 별도 서명 검증이 요구되는지 여부
* **RLS 적용 테이블**: 이 API의 수행 결과 데이터베이스에서 RLS 정책을 거쳐가는 주요 대상 테이블명

## 6. Audit Log 기록 대상 주요 API 목록

다음 10가지 범주의 API는 비즈니스 변경 및 정보 보호에 치명적인 영향을 주는 핵심 API이므로 백엔드 단에서 작업 처리 성공 시 무조건 `Audit Log`를 연동하여 영구 저장한다.
1. **권한 변경 API**: 사용자 권한 템플릿 변경 및 예외 권한 수정 API.
2. **사용자 제어 API**: 신규 임직원/외부 사용자 초대, 휴면/비활성화 처리 API.
3. **민감 개인정보 조회 API**: 마스킹되지 않은 원본 전화번호, 이메일, 주소를 획득하는 API.
4. **기밀 파일 다운로드 API**: Restricted 또는 Confidential 등급 첨부파일 다운로드 서명 발급 API.
5. **대량 내보내기 API**: 엑셀/CSV 데이터 다운로드 API (50건 이상).
6. **FUND/M&A/HR 변경 API**: 지분율 변경, 자본금 추가, 인수 금액 수정, 급여 및 평가 기록 등록 API.
7. **마스터 데이터 수정 API**: NETWORKS 워크스페이스 내의 스타트업/전문가 마스터 병합 및 일괄 업데이트 API.
8. **파기 및 복구 API**: 개인정보 완전 파기(Physical Delete) 및 soft deleted 상태 복원 API.
9. **임시 게스트 공유 API**: 1회성 게스트 뷰 페이지 생성, OTP 전송 및 강제 만료 API.
10. **Service Role 배치 API**: 어드민 전용 스케줄러 배치 구동 및 수동 트리거 실행 API.

## 7. 파일 다운로드 API 계약 표준

* **정적 경로 금지**: 첨부파일 저장 주소(S3 URL 등)를 클라이언트 상에 고정 노출하여 누구나 접근하도록 구현하는 것을 절대 금지한다.
* **Signed URL 요청 계약 흐름**:
  1. 클라이언트는 파일 고유 ID와 함께 다운로드 API 호출.
  2. 서버는 요청자가 해당 파일의 등급에 맞는 권한(`read` 등)을 보유하고 있는지 DB RLS와 세션을 통해 확인.
  3. 검증 성공 시 AWS S3/Storage API를 활용하여 **유효기간 5분**짜리 Presigned URL을 즉시 생성하여 응답 데이터로 반환.
  4. 만약 민감 등급 파일일 경우, 프런트엔드에서 입력받은 [다운로드 사유]가 API Payload에 포함되어 있어야 하며 이를 감사 로그 테이블에 함께 기록한 뒤 Signed URL을 반환한다.
* **외부 사용자 필터**: 외부 스타트업 및 전문가는 본인이 직접 업로드했거나 소속 사업에 배정되어 사전에 배포 승인된 파일 외의 임의 파일 다운로드 요청 시 `permission_denied` 에러를 반환받는다.

## 8. Export API 계약 표준

* **전용 가드 적용**: Export 요청이 들어오면 `checkUserPermission`을 통해 사용자가 일반 쓰기 권한 외에 별도의 `export` 작업 권한이 활성화되어 있는지 확인한다.
* **마스킹 자동화**: 내보내기 엑셀 파일을 백엔드에서 생성할 때, 개인정보(전화번호 등) 열은 자동으로 마스킹 유틸리티가 반영된 텍스트로 치환하여 바이너리를 작성한다.
* **사유 입력 파라미터 필수화**: API 요청 body에 `export_reason` 문자열 필드가 필수적으로 포함되어야 하며, 최소 10자 이상 전달되지 않으면 `validation_error`를 발생시킨다.
* **파일 삭제 예약**: 생성된 파일은 백엔드 임시 캐시 디렉토리에 보관되며 1시간 내 삭제 트리거를 예약하여 파일이 서버 디스크에 장기 방치되는 리스크를 예방한다.

## 9. 입력 검증 및 보안 코딩 표준 (Input Validation)

* **Schema Validation 사용**: 모든 API 및 서버 액션의 인입 지점(Entry Point)에서 **Zod** 또는 동등 수준의 유효성 검사 라이브러리를 사용하여 입력 값의 타입, 길이, 필수 항목 존재 여부를 전량 검증한다.
* **클라이언트 데이터 불신**: 화면 단에서 데이터의 유효성을 검증했더라도 서버에서 이중 검증을 거친다.
* **IDOR (Insecure Direct Object Reference) 방지**:
  * API 요청 시 전달받은 `uuid` 형태의 식별자만을 근거로 조회/수정 로직을 태워서는 안 된다.
  * 해당 리소스를 소유한 기업 ID나 부서 ID가 현재 요청 사용자의 세션 클레임에 정의된 Scope 영역(예: `company_id`)과 정확히 일치하는지 `WHERE` 조건에 교차 대조하여 강제 격리한다.
* **SQL Injection 방지**: ORM(Supabase Client, Prisma 등)의 기본 Parameterized Query를 사용하여 원시 SQL 조합을 제거한다.
* **업로드 파일 검증**: 파일 업로드 API는 파일의 확장자뿐 아니라 MIME-Type을 검증하여 웹셸 등 서버 공격 스크립트(.sh, .php, .js)의 업로드를 백엔드 단에서 원천 차단하고 업로드 용량 제한을 강제한다.

## 10. AC Program 모듈 서버 액션 계약 추가 기준

AC Program First 모듈의 상태 변경은 클라이언트 직접 update가 아니라 서버 액션 또는 RPC에서 검증한 뒤 처리한다. 특히 아래 액션은 Program/Module/Assignment/Guest scope를 모두 확인하고, 성공 시 감사 로그를 남기는 것을 기본값으로 한다.

### 10.1 필수 서버 액션 범위

```txt
신청서 제출 및 회수
서면평가/대면평가/데모데이 평가 제출
심사위원/멘토/상담 전문가 배정
멘토링 관계 생성 및 상담일지 제출
비즈니스 매칭 예약/변경/취소
AI 추천 배정안 확정
출석 체크 및 노쇼 처리
데모데이 투자자 관심 기록
Program KPI 재계산
통합 Excel 다운로드 생성
```

### 10.2 AC 공통 요청 메타데이터

AC 모듈 API 문서에는 다음 메타데이터를 명시한다.

```txt
program_id
program_module_id
participant_id 또는 assignment_id
guest_allowed 여부
required_workspace_permission
required_program_role
audit_log_required 여부
rls_target_tables
```

### 10.3 공통 에러 코드 확장

| 에러 코드 (`code`) | HTTP 매핑 | 적용 정의 |
| :--- | :---: | :--- |
| **`program_scope_denied`** | 403 | 현재 사용자가 해당 Program 또는 Program Module 범위에 접근할 수 없음 |
| **`assignment_scope_denied`** | 403 | GUEST 사용자가 자신에게 배정되지 않은 평가/상담/일정에 접근함 |
| **`state_transition_denied`** | 409 | 현재 상태에서 허용되지 않는 상태 전이를 요청함 |
| **`schedule_conflict`** | 409 | 동일 참가자, 전문가, 장소, 테이블의 시간 충돌이 감지됨 |
| **`export_reason_required`** | 400 | AC 통합 다운로드 요청에 사유가 누락되었거나 기준 길이를 충족하지 않음 |
