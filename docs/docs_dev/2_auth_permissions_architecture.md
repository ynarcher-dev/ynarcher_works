# [2] 인증 및 권한 아키텍처 설계서

## 1. 문서 목적 및 개요

본 문서는 와이앤아처 통합 Works 플랫폼의 사용자 인증, 권한 관리, 데이터 접근 범위 제어 체계에 대한 기술 표준을 정의한다. 

시스템의 권한은 단순한 역할 기반 제어(RBAC)를 넘어 다음 3가지 요소의 조합을 통해 최종적으로 판별되고 강제된다:
```txt
User Role + Workspace Permission + Data Scope
```
* **User Role (사용자 역할)**: 사용자의 신분 및 시스템상의 기본 등급.
* **Workspace Permission (워크스페이스 도메인 권한)**: 특정 비즈니스 도메인(예: AC, FUND, M&A 등)에 대한 읽기/쓰기 권한 제어.
* **Data Scope (데이터 접근 범위)**: 동일한 워크스페이스 내에서 개별 사용자가 조회하거나 조작할 수 있는 데이터의 한계 영역.

## 2. 사용자 유형 정의 (User Roles)

플랫폼을 이용하는 사용자는 다음과 같은 역할을 부여받으며, 각 역할별 보안 위험 요소에 맞춰 최소한의 데이터만 노출해야 한다.

| 사용자 유형 | 역할 설명 | 기본 접근 워크스페이스 | 기본 쓰기 여부 | 기본 데이터 범위 | 주요 보안 리스크 |
| :--- | :--- | :--- | :---: | :--- | :--- |
| **최고 관리자** | 플랫폼 전체 인프라, 설정 및 권한, 로그를 통제하는 최상위 운영자 | 전 영역 | O | `global` (전사) | - 권한 오남용 및 계정 탈취 시 시스템 전체 위험<br>- `service role key` 오용 금지 |
| **경영진** | 전사 실적 모니터링 및 전략 수립을 위한 의사결정권자 | 전 영역 (ADMIN 제외) | X | `global` (전사) | - 전사 민감 정보 및 기밀(M&A, FUND)의 대량 조회로 인한 정보 유출 리스크 |
| **경영지원/관리부서** | 인사(HR), 재무, 총무 등 사내 행정 및 경영지원 담당자 | OFFICE, MANAGEMENT, NETWORKS | O | `global` 또는 `department` | - 임직원 급여 정보, 개인정보, 전자결재 등의 민감 자료 노출 및 불법 유출 |
| **AC 사업부** | 스타트업 발굴 및 보육 프로그램 기획/운영 실무자 | OFFICE, STARTUP, AC, NETWORKS | O | `program` (담당 프로그램) | - 심사위원 평가 점수, 전문가 멘토링 비용 등 사업 지출/평가 정보 노출 |
| **투자/FUND 담당자** | 투자 조합 결성, 출자자 명부 관리 및 포트폴리오 관리 실무자 | OFFICE, STARTUP, FUND, NETWORKS | O | `fund` (담당 펀드) | - 출자자(LP) 투자 금액, 지분 정보 등 최고 기밀 자료의 무단 유출 |
| **M&A 담당자** | 인수합병 딜 소싱 및 기업 실사, 비밀유지계약 전담 실무자 | OFFICE, M&A, NETWORKS | O | `global` | - 딜 성사 전 정보 노출 시 시장 파장 및 NDA 위반에 따른 법적 리스크 |
| **프로젝트 담당자** | 글로벌 및 신사업 등 개별 프로젝트/태스크 수행 실무자 | OFFICE, PROJECT, NETWORKS | O | `project` (담당 프로젝트) | - 신규 프로젝트 기밀 노출 및 외부 협업 과정에서의 계정 오용 |
| **외부 스타트업** | 보육 프로그램 참가 기업 또는 투자 포트폴리오사 담당자 | GUEST | O | `company` (본인 기업) | - 타사 보육 데이터 및 자사 평가 내역 등 접근 불가 정보 노출 차단 필요 |
| **외부 전문가** | 심사위원, 멘토, 자문위원 등 사외 위촉 전문가 | GUEST | O | `self` (배정된 세션) | - 평가 점수, 다른 전문가 의견 조회 금지<br>- 본인에게 할당된 세션 외 접근 금지 |
| **임시 게스트** | 단기 실사단, 외부 감사 등 임시 협업이 필요한 사용자 | GUEST | X (일부 승인 시 Δ) | `temporary` | - 만료 기간 이후의 지속적인 세션 유지 및 데이터 불법 백업 위험 |
| **읽기 전용 사용자** | 모니터링 목적의 비실무 직원 또는 임시 관찰자 | 지정된 영역 | X | `self` 또는 `department` | - 쓰기 권한이 없더라도 불법적인 대량 데이터 export/스크래핑 시도 위험 |

## 3. 워크스페이스 도메인 권한 및 규칙

### 3.1 워크스페이스 식별 키
플랫폼의 도메인은 다음과 같이 10가지 권한 키로 격리되어 통제된다.
* `office`, `startup`, `networks`, `ac`, `fund`, `mna`, `project`, `management`, `admin`, `guest`

> [!IMPORTANT]
> 현재 프론트엔드 라우팅 타입(`WorkspaceKey`)은 위 10개 키를 기준으로 동작합니다. 초기 DB enum·permission seed 일부에는 과거 `hub` 키가 남아 있으므로, 운영 전 `office`/`startup`을 DB 권한 원장과 ADMIN 권한 콘솔에 반영하는 마이그레이션이 필요합니다.

### 3.2 권한 단계 정의
각 워크스페이스에 대한 사용자 접근 단계는 3단계로 정의한다.
1. **`none` (접근 불가)**: 해당 도메인에 대한 모든 데이터 조회가 차단된다.
2. **`read` (조회 가능)**: 데이터를 조회/검색할 수 있으나, 생성/수정/삭제 조작은 차단된다.
3. **`write` (생성/수정/승인 가능)**: 데이터를 생성, 수정, 승인할 수 있는 권한을 가진다.

### 3.3 구현 및 보안 원칙
* **상속 규칙**: `write` 권한은 항상 `read` 권한을 기본적으로 상속 및 포함한다.
* **UI 권한 통제**: `read` 권한이 없는 경우, 좌측 사이드바 및 헤더에서 해당 메뉴 아이콘을 아예 숨겨 사용자 경험(UX)을 유지한다.
* **서버 보안 강제**: **UI에서 버튼이나 메뉴를 감추는 행위는 보안이 아니다.** 최종적인 쓰기/조회 차단은 API 미들웨어, 서버 액션 가드, 데이터베이스 RLS(Row Level Security) 수준에서 반드시 강제되어야 한다.

## 4. 데이터 Scope 모델

동일한 도메인 내에서도 부서 배치, 담당 업무에 따라 데이터 접근 반경을 쪼갠다. RLS와 API 가드는 요청자의 Scope를 판별하여 쿼리 범위를 격리해야 한다.

| Scope 구분 | 적용 의미 | RLS/API 적용 예시 |
| :--- | :--- | :--- |
| **`none`** | 어떠한 데이터에도 접근할 수 없음 | 권한 정지/퇴사자 계정 |
| **`global`** | 플랫폼에 등록된 전사 전체 데이터를 조회 및 조작 가능 | 최고 관리자, 경영진 |
| **`department`** | 사용자가 속한 부서(조직)에 할당된 데이터만 접근 가능 | 경영지원 부서, 인사 실무자 |
| **`program`** | 사용자가 담당자로 배정된 AC 보육 프로그램 및 그 신청 기업 정보만 접근 가능 | AC 사업부 담당 매니저 |
| **`project`** | 본인이 멤버로 등록된 특정 프로젝트 및 태스크만 접근 가능 | 프로젝트 실무자, 협업 스태프 |
| **`fund`** | 본인이 관리자 또는 책임자로 소속된 조합(펀드)의 회계/출자 데이터만 접근 가능 | 투자실 담당 심사역 |
| **`company`** | 로그인한 계정이 소속된 외부 스타트업(기업) 관련 데이터에 한해서만 접근 | 외부 스타트업 담당자 |
| **`self`** | 본인이 직접 생성했거나 멘토링/심사역으로 배정된 단일 레코드만 접근 가능 | 외부 전문가(멘토), 심사위원 |
| **`temporary`** | 만료 예정 기한(`expires_at`) 정보가 추가된 한시적인 범위에 한하여 임시 접근 | 임시 게스트, 단기 자문단 |

## 5. 권한 템플릿 매트릭스 (기본형)

시스템 초기 생성 및 시드 데이터를 위한 사용자 유형별 기본 권한 매트릭스는 다음과 같다. 관리자는 `admin` 워크스페이스 내 권한 설정 UI를 통해 해당 디폴트 값을 수정할 수 있어야 한다.

| 사용자 유형 | OFFICE | STARTUP | NETWORKS | AC | FUND | M&A | PROJECT | MANAGEMENT | ADMIN | GUEST |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| **최고 관리자** | RW | RW | RW | RW | RW | RW | RW | RW | RW | - |
| **경영진** | R | R | R | R | R | R | R | R | - | - |
| **경영지원** | RW | R | RW | R | R | - | RW | RW | - | - |
| **투자실** | R | RW | RW | R | RW | - | RW | R | - | - |
| **사업부** | R | RW | RW | RW | - | - | RW | R | - | - |
| **M&A팀** | R | R | RW | - | - | RW | RW | - | - | - |
| **프로젝트팀** | R | R | RW | R | - | - | RW | R | - | - |
| **외부 스타트업** | - | - | - | - | - | - | - | - | - | RW |
| **외부 전문가** | - | - | - | - | - | - | - | - | - | RW |
| **임시 게스트** | - | - | - | - | - | - | - | - | - | R |
| **읽기 전용 사용자** | R | R | R | R | R | R | R | R | - | - |

* `R` = Read-only (읽기 권한 활성화)
* `RW` = Read/Write (읽기 및 쓰기 권한 모두 활성화)
* `-` = None (접근 권한 없음)

## 6. 임시 권한 관리 정책

외부 감사, 단기 실사단, 또는 한시적 협업을 위해 발급되는 게스트/임시 권한은 보안 유출의 가장 큰 취약점이므로 다음 규정을 강제 적용한다.
1. **만료 기한 필수 지정**: 모든 임시 권한 테이블에는 `expires_at` 필드가 필수적으로 존재해야 하며, 생성 시 최대 기한(예: 30일)을 넘길 수 없다.
2. **실시간 차단 로직**: 
   * UI: 만료 일자가 지난 경우 대시보드 진입을 차단하고 "만료된 권한입니다" 메시지를 노출한다.
   * API & RLS: 서버 및 DB 쿼리 실행 시 `now() < expires_at` 조건을 검사하여 토큰 세션이 유효하더라도 권한 판단에서 즉시 실패하도록 설계한다.
3. **감사 로그 대상**: 임시 권한을 수동으로 부여하거나 연장, 회수하는 모든 행위는 `Audit Log` 테이블에 사유(`reason`)와 함께 무조건 기록되어야 한다.

## 7. 권한 변경 감사 로그 (Audit Log)

사용자의 권한 그룹이 변경되거나 워크스페이스별 예외 권한이 설정될 경우, 아래 항목을 수집하여 수정 불가능한 감사 로그 테이블에 영구 저장한다.
* `actor_user_id` (조작을 수행한 관리자 고유 ID)
* `target_user_id` (권한이 변경되는 대상 사용자 고유 ID)
* `changed_workspace` (변경된 워크스페이스 식별자)
* `before_permission` (변경 전 권한 상태: `none`, `read`, `write`)
* `after_permission` (변경 후 권한 상태: `none`, `read`, `write`)
* `reason` (권한 변경 사유 - 한글 10자 이상 입력 필수)
* `created_at` (로그 기록 일시)
* `request_ip` (조작 요청이 들어온 클라이언트 IP 주소)
* `user_agent` (조작 관리자의 브라우저/OS 접속 환경 정보)

## 8. 개발자를 위한 아키텍처 구현 기준

개발자는 위 권한 사양에 맞춰 아래 구성 요소를 구현해야 한다.
1. **공통 권한 Helper**: 현재 세션 사용자 객체를 파싱하여 특정 워크스페이스 권한 및 Scope 정보를 추출하는 유틸리티 함수(예: `checkUserPermission(session, workspace_key)`) 구현.
2. **API 권한 Guard**: Supabase Edge Function 또는 RPC 함수 진입점 단계에서 API 실행 권한을 사전 검증하여 부적절한 요청에 대해 표준 에러 객체(`permission_denied`) 반환.
3. **RLS Helper SQL**: PostgreSQL DB 상에서 현재 JWT 클레임 정보를 기반으로 사용자의 Scope와 권한 레벨을 반환하는 SQL 함수 작성.
4. **권한 템플릿 Seed**: 최초 데이터베이스 마이그레이션 단계에서 11개 사용자 유형별 권한 매트릭스 정보를 시드 데이터로 주입.
5. **권한 변경 Audit Log Trigger**: 권한 관련 테이블에 `INSERT` 또는 `UPDATE` 발생 시 자동으로 감사 로그 테이블에 데이터가 적재되도록 RLS 정책 및 트리거 설정.
6. **본인 권한 조회 API**: 클라이언트 App 구동 시 최초 1회 본인의 권한 및 적용 범위 스냅샷을 안전하게 획득할 수 있는 REST/RPC API를 제공하되, 화면 렌더링 최적화 목적으로만 사용하고 실제 보안 통과용으로 신뢰하지 않는다.

## 9. 보안 설계 체크리스트

> [!NOTE]
> AC Program First 모듈은 단순 `AC read/write` 권한만으로 보호하지 않고, `program_id`, `program_module_id`, `assignment_id`, `participant_id`, `company_id` 범위를 함께 검증한다. 내부 운영자는 담당 Program 범위로 제한하고, 외부 스타트업/심사위원/멘토/투자자는 GUEST 전용 화면에서 자기 회사, 자기 배정, 자기 일정만 처리한다.

모든 기능 개발 시 개발자는 다음 체크리스트를 검토하여 누락이 없음을 보장해야 한다.
* [ ] 본 기능(또는 화면)을 사용하기 위해 요구되는 워크스페이스 권한이 정확히 매핑되었는가?
* [ ] 읽기와 쓰기 분기가 완벽히 처리되어 쓰기 권한이 없는 사용자는 생성/수정 버튼이 서버 수준에서 차단되는가?
