# [7] 데이터베이스 설계 및 테이블 통합 가이드라인

본 문서는 **와이앤아처 통합 Works 플랫폼**의 데이터베이스(Supabase/PostgreSQL) 설계 시, 테이블이 기능별로 무분별하게 쪼개지는 현상(테이블 단편화)을 방지하고 일관성 있고 효율적인 데이터 모델을 유지하기 위한 구체적인 규칙을 정의합니다.

---

## 📌 1. 가이드라인의 목적
* **테이블 및 마이그레이션 비대화 방지**: 기능마다 개별 테이블을 생성하여 관리 오버헤드가 증가하고 마이그레이션 파일 수가 백여 개 이상으로 팽창하는 안티패턴을 사전에 예방합니다.
* **도메인 모델의 응집도 향상**: 화면이나 UI 중심이 아닌, 실제 비즈니스 도메인(Entity) 중심으로 스키마를 정규화하여 중복 데이터를 방지하고 조인(Join) 복잡도를 낮춥니다.
* **유지보수 용이성 및 RLS 일관성 확보**: 정형화된 스키마 구조를 통해 Row Level Security 정책 적용을 단순화하고 성능 병목을 사전에 제어합니다.

---

## 📐 2. 테이블 설계 3대 대원칙

### 2.1 화면 및 기능 중심의 테이블 생성 금지
* 특정 화면의 레이아웃이나 단일 기능(예: "마이페이지 스크랩 탭", "기업 상세 페이지 정보")에 1:1로 대응하는 테이블을 각각 신규 생성하지 않습니다.
* UI 표현 방식은 프론트엔드의 몫이며, DB는 비즈니스 도메인의 핵심 개념(예: `users`, `startups`, `programs`, `applications`)을 담는 테이블로 응집성 있게 정의되어야 합니다.

### 2.2 1:1 관계의 테이블 통합 (Single-Table Principle)
* 데이터의 생성·소멸 주기(Lifecycle)가 같고 1:1 관계를 지니는 정보는 테이블을 분리하지 않고 **하나의 테이블 내 컬럼으로 통합**하는 것을 원칙으로 합니다.
* **안티패턴 예시**:
  * `startups` (기업 정보) / `startup_addresses` (기업 주소) / `startup_sns_links` (기업 SNS 정보)로 각각 테이블을 쪼개는 방식.
* **올바른 방식**:
  * `startups` 테이블 내에 주소 정보 필드(또는 JSONB)를 컬럼으로 추가하여 1:1 데이터를 한 곳에 관리합니다.

### 2.3 다형적 관계(Polymorphic Association)의 공통 테이블 통합
* 첨부파일, 댓글/메모, 태그, 감사 로그(Audit Log) 등 여러 도메인에서 공통적으로 필요로 하는 보조 성격의 기능은 대상 엔티티별로 테이블을 복제하여 쪼개지 않습니다.
* **안티패턴 예시**:
  * `startup_files` (스타트업 첨부파일), `expert_files` (전문가 첨부파일), `program_files` (프로그램 첨부파일)로 각각 테이블을 생성하는 방식.
* **올바른 방식**:
  * 단일 공통 테이블 `attachments`을 설계하고, `target_type`('STARTUP', 'EXPERT', 'PROGRAM')과 `target_id`(UUID) 복합 구조를 채택하여 하나의 테이블에서 모든 첨부파일을 처리합니다.

## 🔍 3. 물리적 테이블 분리(쪼개기)가 정당화되는 예외 기준

테이블 통합이 기본 원칙이지만, 다음 3가지 예외 기준 중 하나 이상을 충족하는 경우에만 물리적으로 테이블을 분리하여 설계하는 것을 허용합니다.

| 분류 | 분리 허용 사유 | 구체적 적용 사례 |
| :--- | :--- | :--- |
| **보안 격리 및 RLS 차등** | 동일 도메인 내 특정 데이터가 강력한 암호화나 차별화된 RLS 권한 제어를 받아야 하는 경우 | `users` 테이블에서 민감 정보(주민번호 등 식별 정보, 급여 정보)를 `user_private_profiles`로 격리 |
| **물리적 성능 병목 제어<br>(Vertical Partitioning)** | 수십 MB 크기의 대용량 바이너리, 텍스트 데이터 등을 포함하고 있어, 빈번한 메타데이터 인덱스 스캔 시 I/O 병목이 우려되는 경우 | `posts` 테이블의 메타데이터와 대용량 본문 텍스트(`content_html` 등)를 `post_contents`로 분리 |
| **극단적인 쓰기/읽기 주기 차이** | 마스터 정보(Read Heavy)와 실시간 집계/로그 데이터(Write Heavy)가 공존하여 스캔 비용을 절감해야 하는 경우 | `programs` (프로그램 상세 마스터) 정보와 실시간 클릭/접속 추적 데이터를 담는 `program_logs` 분리 |

---

## 🛠️ 4. 유연성 확보를 위한 JSONB 활용 가이드

관계형 데이터베이스의 정적 타입 장점을 살리되, 불필요한 테이블 생성을 줄이기 위해 PostgreSQL의 `jsonb` 타입을 적극 활용합니다.

1. **JSONB 컬럼 권장 대상**:
   * 관계(Foreign Key) 제약 조건이나 조인(Join) 연산이 필요 없는 단순 정보성 데이터.
   * 속성이 수시로 추가되거나 가변적인 비정형 메타데이터 (예: SNS 계정 링크 목록, 개별 UI 레이아웃 설정 값, 복잡하지 않은 유저 환경설정).
2. **JSONB 사용 지양 대상**:
   * 다른 테이블의 Primary Key를 참조하여 무결성(Referential Integrity)을 유지해야 하는 경우.
   * 해당 속성에 대해 잦은 범위 검색, 인덱싱 및 집계(Aggregation) 쿼리가 발생하는 핵심 비즈니스 데이터.

## 📁 5. 마이그레이션 파일 관리 수칙 (Supabase)

마이그레이션 파일 개수가 불필요하게 늘어나는 것을 방지하고 형상 관리를 단순하게 유지하기 위한 규칙입니다.

1. **피처 단위 마이그레이션 병합 (Squashing)**:
   * 로컬 개발 도중 컬럼 추가, 타입 변경 등으로 인해 하루에도 수차례 마이그레이션 파일이 새로 생성될 수 있습니다.
   * 이를 그대로 운영 환경에 배포하면 파일 갯수가 폭증하므로, 기능 개발(`Feature branch`)이 완료되어 상위 브랜치(Staging/Main)에 병합하기 직전에 **해당 피처와 관련된 마이그레이션 파일들을 하나로 병합(Squash)**하여 반영합니다.
2. **마이그레이션 스크립트 작성 표준**:
   * 모든 DDL(스키마 변경) 스크립트는 여러 번 실행해도 에러가 나지 않는 **멱등성(Idempotency)**을 가지도록 설계합니다.
   * 테이블 생성: `CREATE TABLE IF NOT EXISTS table_name (...);`
   * 컬럼 추가: `ALTER TABLE table_name ADD COLUMN IF NOT EXISTS column_name data_type;`
   * 인덱스 생성: `CREATE INDEX CONCURRENTLY IF NOT EXISTS index_name ON table_name (column_name);`
