# [5] 백업, 보존 및 개인정보 파기 운영 기준서

## 1. 문서 목적 및 개요

본 문서는 와이앤아처 통합 Works 플랫폼 데이터베이스와 파일 저장소의 안정적인 백업 및 신속한 복구 절차를 수립하고, 법적 의무 사항(개인정보보호법 및 상법/세법 등)을 만족하기 위한 데이터 보존 기간 및 개인정보 파기 기준을 정의한다. 

이를 통해 서비스의 지속 가능성을 보장하고, 법적 리스크를 통제한다.

---

## 2. 운영 핵심 원칙

1. **이력 데이터 보존**: 업무 수행 과정에서 발생한 핵심 비즈니스 이력 데이터는 필요한 기간 동안 안전하게 보존하며 파편화를 방지한다.
2. **목적 달성 후 파기**: 수집된 개인정보는 동의받은 수집 목적을 달성하거나 보존 기간이 경과하면 복구할 수 없는 방식으로 지체 없이 파기 및 익명화한다.
3. **Soft Delete 기본 적용**: 일반 업무 조작 과정에서의 레코드 삭제는 데이터 복원과 히스토리 추적을 위해 물리적 삭제가 아닌 논리적 삭제(`is_deleted = true`)를 기본으로 적용한다.
4. **제한적 Physical Delete**: 법적인 파기 요구서 접수 등 특수 상황 발생 시 상위 승인을 거쳐 데이터베이스에서 데이터를 물리적으로 완전 삭제(`DELETE` 실행)한다.
5. **정기 복구 검증**: 백업 파일의 존재 유무 확인을 넘어, 백업된 데이터를 사용해 정상적으로 서비스를 구동할 수 있는지 정기 검증 리허설을 실시한다.
6. **Staging 환경 운영 정보 반입 금지**: 테스트 및 스테이징 환경에는 운영 환경의 실제 개인정보가 포함된 DB 덤프 파일을 그대로 복원하여 사용해서는 안 된다. 개인정보는 필수로 마스킹되거나 난수화(Pseudonymization)된 데이터만 로드한다.
7. **감사 로그 수정 금지**: `Audit Log` 데이터는 시스템 운영자나 최고 관리자라 할지라도 직접적인 수정 및 삭제가 원천 불가능하도록 보관한다.

## 3. 데이터 백업 기준

시스템의 가용성과 데이터 무결성을 위해 백업 정책을 정기적으로 수립하여 관리한다.

### 3.1 백업 대상 목록
* **데이터베이스**: Supabase/PostgreSQL 전체 스키마 및 레코드 데이터.
* **파일 저장소**: AWS S3/Supabase Storage 내에 저장된 첨부파일 전체.
* **파일 메타데이터**: 파일의 버저닝 정보 및 경로 매핑 DB 데이터.
* **원본 Import 파일**: 외부에서 엑셀 대량 임포터 등을 통해 주입된 원천 로우 데이터 아카이브.
* **마이그레이션 소스**: 데이터베이스 마이그레이션 SQL 스크립트.
* **릴리즈 기록**: 애플리케이션 빌드 아티팩트 및 릴리즈 정보.
* **시스템 중요 설정**: 환경 변수 템플릿(비밀 값 제외) 및 Nginx, 인프라 라우팅 정보.

### 3.2 백업 제외 민감 데이터
* 백업 데이터 덤프 내에 `service_role` 키, `access_token`, `refresh_token`, 각 인증 API Secret 키 원문 등 복구 세션 유출에 치명적인 데이터는 포함하지 않고, 필요 시 복구 완료 후 신규 인출하도록 처리한다.

## 4. 재해 복구(DR) 기준 및 복구 테스트

시스템 장애 발생 시 비즈니스 연속성 계획(BCP)에 의거하여 다음과 같은 목표 시점 및 목표 시간을 설정하여 운영한다.

### 4.1 서비스 복구 목표 기준

| 항목 | 정의 | 권장 복구 기준 |
| :--- | :--- | :--- |
| **RPO** (Recovery Point Objective) | 데이터 손실 허용 시점 (마지막 백업 후 유실되는 데이터의 한계 시점) | **24시간 이내** (최대 일 단위 스냅샷 백업 제공) |
| **RTO** (Recovery Time Objective) | 서비스 복구 목표 시간 (장애 발생 시점부터 서비스 재개까지 소요되는 최대 시간) | **주요 서비스 4시간 이내** |

### 4.2 분기별 복구 리허설 계획
시스템 관리자는 분기 1회 스테이징/테스트 환경을 활용하여 아래 복구 검증 절차를 수행하고 결과를 문서로 기록하여야 한다.
1. **백업 본 덤프 다운로드**: 자동 예약 백업 저장소로부터 최신 덤프 파일 획득.
2. **스테이징 DB 복원**: 격리된 스테이징 데이터베이스 인스턴스에 복원 쿼리 수행.
3. **데이터 정합성 검증 (Row Count)**: 마스터 테이블과 주요 업무 테이블의 데이터 건수를 원본 서버와 대조하여 유실 유무 확인.
4. **시스템 로그인 검증**: 테스트 사용자 계정으로 복원된 디비 세션에 로그인이 성공적으로 통과되는지 확인.
5. **RLS Smoke Test**: 복원된 디비 상에서 RLS 규칙이 깨지지 않고 정상 작동하여 권한 격리가 이루어지는지 검증 SQL 스크립트 실행.
6. **첨부파일 접근 검증**: 저장소 복원 후 만료된 Signed URL 발급 시 다운로드가 원활하게 동작하는지 확인.

## 5. 데이터 보존 기간 기준

> [!NOTE]
> 본 보존 기간은 와이앤아처의 내부 효율성 및 법적 이력 유지를 고려한 초기 제안 스펙이다. 실제 운영 및 외부 감사 규정 적용 시 **최종 법무/회계 검토**를 거쳐 보존 정책을 확정해야 한다.

| 데이터 구분 | 권장 보존 기간 | 보존 근거 및 세부 규정 |
| :--- | :--- | :--- |
| **마스터 기본 정보** | 업무상 해당 기업/전문가 관계가 활성화된 기간 | NETWORKS 디렉토리 상의 스타트업 및 전문가 원천 이력 정보 |
| **프로نامج 신청/평가 데이터** | 해당 보육 사업 종료 후 최대 5년 | 정부 보육 사업 지원비 정산 및 감사 증빙 자료용 |
| **투자/FUND 자료** | 조합 해산 및 청산 완료 후 5년 ~ 10년 | 상법 제33조(상업장부 등의 보존) 및 세법 기준 의무 보존 |
| **M&A 딜 및 협상 문서** | 딜 종결(성공/실패) 후 최소 5년 | 분쟁 대비 및 영업비밀보호법 공소시효 등 대응용 |
| **HR 기록 및 급여 데이터** | 퇴사 후 3년 (일부 기초 서류 3년) | 근로기준법 제42조(계약 서류의 보존) 및 퇴직급여 보존 |
| **감사 로그 (Audit Log)** | 최소 3년 이상 (5년 권장) | 개인정보보호법 고시 및 금융 규제 보안 로그 의무 보존 |
| **Import Raw Data (엑셀 원본)** | 마스터 주입 및 검증 완료 후 3개월 | 업로드 정합성 오류 수정 검증이 끝난 후 즉시 아카이브 처리 및 파기 |
| **임시 게스트 데이터** | 공유 목적 달성 즉시 또는 최대 30일 | 게스트에게 발급된 임시 페이지 및 접속 로그는 목적 종료 시 파기 |

## 6. 개인정보 파기 및 익명화 처리 지침

보존 기간이 만료되거나 정보 주체의 동의 철회 및 파기 요청이 있을 경우, 아래의 처리 방식을 기준으로 적용한다.

### 6.1 개인정보 처리 방식 정의

| 방식 | 의미 | 적용 업무 및 대상 |
| :--- | :--- | :--- |
| **`Soft Delete`** | 데이터베이스 레코드에 `is_deleted = true` 플래그만 마킹하여 화면 상 노출을 중단하고 논리적으로 제거함. | 일반 임직원이 업무 화면에서 '삭제' 버튼을 눌렀을 때의 디폴트 처리 |
| **`Masking`** | 데이터의 특정 문자 열을 비식별 기호(`*`)로 치환하여 일부 정보를 숨김. | 조회 권한자가 아닌 일반 사원용 목록 뷰 및 통계 리포트 출력 시 |
| **`Anonymization`** | 데이터 복원이 절대 불가능하도록 이름, 전화번호 등의 개인 식별 키를 삭제하고 통계적 형태로 변형함. | 보육 통계 이력은 유지하되, 보존 기한이 지난 스타트업 대표 정보 파기 시 |
| **`Physical Delete`** | 데이터베이스 테이블에서 레코드 행(Row) 자체를 물리적으로 영구 삭제(`DELETE` 문 실행)하여 복구가 불가능하게 만듦. | 정보주체의 법적 파기 요구 접수 및 법률적 의무 보존 기한이 끝난 개인정보 |

### 6.2 삭제 요청 처리 프로세스 (7단계)
사용자 및 외부 스타트업으로부터 개인정보 삭제(또는 탈퇴) 요청이 공식 접수되었을 때 시스템 관리자는 다음 절차를 따른다.

```txt
1. [요청자 신원 확인] ➔ 2. [대상 데이터 식별] ➔ 3. [법적/업무상 보존 필요 여부 검토] ➔ 4. [처리 방식 결정] ➔ 5. [승인 및 처리 기록 생성] ➔ 6. [데이터 파기 수행] ➔ 7. [결과 피드백 통보]
```

1. **[1단계] 요청자 신원 확인**: 본인 신분증 사본 접수 또는 휴대폰/이메일 OTP 인증을 통한 동일인 검증.
2. **[2단계] 대상 데이터 식별**: 탈퇴 요청한 계정에 매핑된 NETWORKS, AC, FUND, MANAGEMENT 내 전체 개인정보 레코드 취합.
3. **[3단계] 법적/업무상 보존 필요 여부 검토**: 상법, 국세기본법, 근로기준법 등에 의해 의무 보존해야 하는 투자/지분/급여 계약 내역이 존재하는지 확인.
4. **[4단계] 처리 방식 결정**: 보존 의무가 없는 정보는 `Physical Delete` 및 `Anonymization`을 진행하고, 법적 보존 의무가 남은 데이터는 분리 보관 및 조회 불가 처리.
5. **[5단계] 승인 및 처리 기록 생성**: 개인정보 관리책임자의 전자결재 승인을 득하고, 작업 예정 내역을 보존 기록 문서에 등록.
6. **[6단계] 데이터 파기 수행**: 백엔드 파기 스크립트를 사용하여 데이터베이스에서 로우 삭제 및 AWS S3 내 업로드 파일 물리 삭제 처리.
7. **[7단계] 결과 피드백 통보**: 신청자에게 파기 처리가 완료되었음을 공식 이메일 또는 문자메시지로 결과 발송.

## 7. Audit Log 보존 및 관리 정책

1. **로그 영구 보존 원칙**: 권한의 불법 획득 및 기밀 유출 행위의 원인 파악을 위해 수집된 아래 감사 로그 대상 작업 내역은 수정 및 삭제가 원천 불가하며 최소 3년 이상 보관한다.
   * 사용자 권한 그룹 변경
   * 제한(Restricted) 등급 및 개인정보 원본 조회 내역
   * 기밀 첨부파일 다운로드 이력
   * 대량 Excel/CSV 데이터 내보내기(Export)
   * `service_role`을 통한 시스템 강제 조작 기록
   * 운영 장애 대응 긴급 핫픽스 패치 기록
   * NETWORKS 마스터 데이터 강제 병합 및 일괄 변경 이력
2. **로그 적재 시 주의사항**: 감사 로그 테이블(`audit_logs`) 내의 세부 내용 필드에 주민등록번호 원문, 이메일 비밀번호, API Secret Key 등 또 다른 민감 정보가 텍스트 형식으로 포함되지 않도록 마스킹 처리 후 저장하여, 감사 로그 자체가 개인정보 침해 요인이 되지 않도록 제어한다.
