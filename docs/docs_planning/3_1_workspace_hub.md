# [3-1] OFFICE (전사 업무 허브, 구 HUB) 워크스페이스 상세 기능 요건서

본 문서는 전사 임직원이 매일 진입하는 업무 허브인 `OFFICE` 워크스페이스의 화면 요건과 비즈니스 흐름을 정의합니다. 기존 문서 체계의 `HUB`는 통합 조회 센터였으나, 현재 구현에서는 대시보드·AI 에이전트·임직원 정보·전사 캘린더·전자결재·게시판이 `OFFICE`로 통합되었습니다.

> [!NOTE]
> 파일명 `3_1_workspace_hub.md`는 기존 링크와 참조를 보존하기 위해 유지합니다. 본문에서 말하는 정본 워크스페이스명은 `OFFICE`이며, 코드 라우트는 `/office`, 권한 키는 `office`입니다.

---

## 1. 주요 화면 및 기능 명세

### 1.1 OFFICE 대시보드 (Dashboard)
* **목적**: 전사 핵심 현황과 자주 쓰는 업무 진입점을 한 화면에 모아, 임직원이 로그인 직후 업무 맥락을 빠르게 파악하도록 합니다.
* **구성 요소**:
  * **개인화 인사 영역**: 사용자명과 입사일 기준 근속 일수(D+n)를 상단 중앙에 노출합니다.
  * **통합 검색 패널**: 스타트업, 임직원, 프로그램, 프로젝트, 딜, 펀드 등 전사 주요 객체를 한 번에 검색하는 진입점입니다.
  * **워크스페이스 대표 지표 카드**: 임직원 정보, 스타트업 네트워크, 투자/전문가 네트워크, 협력사 네트워크, 사업 현황, 글로벌/신사업 현황, M&A/PE 현황, 투자현황, 경영현황을 카드형 딥링크로 제공합니다.
  * **공유 게시판 미리보기**: 공지사항, 공용자료실, 인사이트 게시판의 최근 글을 우측 컬럼에 요약 노출합니다.
  * **AI 에이전트 배너**: 자연어 기반 검색/분석 에이전트로 이동하는 CTA를 제공합니다.

### 1.2 AI 에이전트 (AI Agent)
* **목적**: 사용자가 자연어로 스타트업, 투자, M&A, 프로젝트, 전사 일정 등에 대한 질문을 입력하고, 요약 리포트와 관련 딥링크를 받는 지능형 검색·분석 채널입니다.
* **현재 구현 상태**: OFFICE 사이드바의 `AI 에이전트` 탭과 전용 패널 UI가 준비되어 있습니다.
* **후속 확정 사항**:
  * **연동 방식**: RAG(Retrieval-Augmented Generation)와 Text-to-SQL 중 실제 운영 보안에 적합한 방식을 확정합니다.
  * **보안 원칙**: AI 응답은 사용자의 워크스페이스 권한과 데이터 Scope를 초과할 수 없으며, 출처 딥링크도 접근 가능한 객체에만 생성합니다.
  * **감사 로그**: 민감 도메인(M&A/FUND/인사) 질의는 질의문, 응답 출처, 사용자 ID, 시각을 감사 로그 또는 별도 관측 로그로 남기는 방안을 검토합니다.

### 1.3 임직원 정보 (Staff Directory)
* **목적**: MANAGEMENT의 임직원 원장 데이터를 읽기 전용으로 재사용하여, 전사 구성원과 심사역 정보를 안전하게 조회합니다.
* **주요 요건**:
  * **조회 전용 재사용**: `EmployeeDirectory`를 `readOnly` 모드로 사용하며, OFFICE에서는 비활성화/수정 액션을 노출하지 않습니다.
  * **상세 이동**: 목록 행 클릭 시 `/office/managers/:id` 경로의 읽기 전용 상세 페이지로 이동합니다.
  * **마스킹**: 이메일 및 연락처는 앱 계층 마스킹 정책을 유지합니다.
  * **원장 원칙**: 이름, 역할, 부서, 프로필 등 임직원 원본 수정은 MANAGEMENT 인사 관리 또는 마이페이지의 허용 컬럼을 통해서만 수행합니다.

### 1.4 전사 캘린더 (Company Calendar)
* **목적**: AC, PROJECT, FUND, 사내 일정 등 여러 워크스페이스에서 발생하는 주요 일정을 한 화면에서 확인합니다.
* **레이어 구성**:
  * **AC 일정**: 프로그램 기간, 모집 마감, 평가, OT, 멘토링, 데모데이 등 사업 운영 일정.
  * **PROJECT/M&A 일정**: 프로젝트 마일스톤, 딜 단계 주요 기한, 외부 협약·IR 일정.
  * **FUND 일정**: 캐피탈 콜 납입 요청 기한 및 주요 펀드 운용 일정.
  * **사내 일정**: 전사 행사, 주요 결재 기한, 경영지원 공지성 일정.
* **연동 원칙**: 일정 원장은 각 도메인 테이블과 `system_events`에 분산될 수 있으나, OFFICE 캘린더는 조회 오버레이 역할만 수행합니다.

### 1.5 전자결재 (Approval)
* **목적**: 기존 전자결재 워크스페이스성 기능을 OFFICE 안으로 이관하여, 전사 공통 업무 흐름 안에서 결재 문서를 조회하고 처리합니다.
* **현재 구현 상태**: `OFFICE > 전자결재` 탭에서 `ApprovalTable`을 렌더링합니다.
* **업무 원칙**:
  * **상태 머신**: 대기, 진행, 승인, 반려 등 결재 상태는 서버 데이터와 상태 머신 기준을 따릅니다.
  * **경영지원 연계**: 결재 승인 금액은 MANAGEMENT의 재무·예산 대시보드와 연결됩니다.
  * **알림 연계**: 알림톡/SMS/이메일 실발송은 배포·운영 단계의 알림 채널 설정에 따릅니다.

### 1.6 게시판 및 자료 공유 (Boards)
* **목적**: 공지사항, 공용자료실, 인사이트 등 전사 공유 콘텐츠를 OFFICE 안에서 생성·열람합니다.
* **구성 요건**:
  * **고정 게시판**: 공지사항, 공용자료실 등 항상 상단 그룹에 노출되는 게시판입니다.
  * **일반 게시판**: 인사이트 및 ADMIN 게시판 관리에서 추가되는 신규 게시판입니다.
  * **동적 주입**: 사이드바는 게시판 레지스트리를 기반으로 고정 게시판과 일반 게시판을 평탄하게 나열합니다.
  * **작성자 표시**: 게시글 작성자명, 작성일, 신규 표시(NEW)를 목록에서 확인할 수 있습니다.

### 1.7 회의실 예약 및 거래처 정보 (후속 골격)
* **회의실 예약**: OFFICE 사이드바의 `회의실 예약` 탭과 준비 중 화면 골격이 존재합니다. 실제 예약 시간표, 충돌 검증, 회의실 자원 테이블은 후속 구현 대상입니다.
* **거래처 정보**: OFFICE 사이드바의 `거래처 정보` 탭과 준비 중 화면 골격이 존재합니다. 거래처 원장 위치, NETWORKS와의 중복 범위, 전자결재/정산 연계 규칙은 후속 정합화 대상입니다.

## 2. 데이터 연동 규격 (Database / Store Mapping)

| 화면 영역 | UI 구성 요소 | 참조 원천 | 주요 연동 항목 | 비즈니스 룰 |
| :--- | :--- | :--- | :--- | :--- |
| 대시보드 | 인사/네트워크/사업/투자/경영 요약 카드 | `useHubSummary`, 각 도메인 집계 | 임직원 수, 스타트업 수, 전문가/협력사 수, 프로그램/프로젝트/딜/펀드 지표 | 카드 클릭 시 원 도메인 워크스페이스로 딥링크 |
| 대시보드 | 통합 검색 | `UnifiedSearchPanel` | 스타트업, 전문가, 임직원, 프로그램, 프로젝트 등 | 검색 결과는 사용자 권한을 넘지 않도록 필터링 |
| AI 에이전트 | 자연어 질의 패널 | `AiAgentPanel` | 질의문, 추천 딥링크, 요약 응답 | RAG/Text-to-SQL 방식 확정 전까지 UI 중심 |
| 임직원 정보 | 읽기 전용 디렉토리 | `users`, `departments`, `dept_members`, `org_versions` | 이름, 이메일, 연락처, 역할, 조직 경로 | 수정은 MANAGEMENT/마이페이지 허용 범위에서만 수행 |
| 전사 캘린더 | 월간/일정 패널 | `system_events` 및 도메인별 일정 | 제목, 시작/종료일, source_type, 레이어 | OFFICE는 조회 오버레이만 담당 |
| 전자결재 | 결재 문서 테이블 | approval 계열 테이블/RPC | 상태, 결재선, 문서 제목, 기안자 | 최종 권한과 상태 전이는 서버 기준 |
| 게시판 | 게시판 홈/글 목록 | `boardStore` 및 게시판 레지스트리 | board tab, 제목, 작성자, 작성일, 고정 여부 | ADMIN 게시판 관리에서 생성된 게시판을 OFFICE에 노출 |

## 4. 권한 및 보안 원칙

* **워크스페이스 권한**: OFFICE 진입은 프론트 라우팅 기준 `RequireWorkspace workspace="office"`를 통과해야 합니다.
* **원장 비소유 원칙**: OFFICE는 전사 조회·업무 허브이며, 스타트업·네트워크·임직원·펀드·딜 원장을 직접 소유하지 않습니다.
* **마스킹**: 임직원 이메일/연락처 등 개인정보는 OFFICE 조회 화면에서도 마스킹 정책을 적용합니다.
* **AI 보안**: AI 에이전트는 UI 수준 숨김을 보안으로 간주하지 않으며, 실제 응답 생성 시 서버/RLS 권한 검사를 반드시 통과해야 합니다.
* **후속 정합화**: 기존 DB enum·permission seed에 남아 있는 `hub` 키와 현재 프론트 `office` 키의 차이는 별도 마이그레이션 및 권한 콘솔 정비 대상입니다.
