# [3-5] FUND (투자실) 워크스페이스 상세 기능 요건서

본 문서는 투자실 담당자가 조합/펀드 결성 및 지분 관리, 투자 포트폴리오 집행, 캐피탈 콜, 그리고 **정기 영업보고·운용보고 및 투자기업 사후관리**를 효율적으로 수행할 수 있도록 화면 요건과 비즈니스 흐름을 정의합니다.

> [!IMPORTANT]
> 본 워크스페이스는 개인투자조합(개인투자조합)의 업무집행조합원(GP)이 산출하는 **공식 보고서(영업보고서·운용보고서)를 시스템에서 자동 생성**하는 것을 최종 목표로 합니다. 따라서 마스터 데이터의 필드 정의는 실제 보고서 서식(모태펀드 온기·반기 영업보고서, 월간 투자현황)을 단일 원천으로 삼습니다.

---

## 1. 주요 화면 및 기능 명세

### 1.1 펀드 현황 보드
* **결성 조합 및 잔액 상태 (종합 대시보드)**
  * **기능**: 운용 중인 모든 투자 조합(펀드)의 총액과 실시간 잔액을 직관적으로 모니터링합니다.
  * **핵심 지표**:
    * **총 결성액(약정총액)**: 최초 펀드 결성 시 약정된 금액.
    * **누적 투자 집행액**: 해당 펀드에서 포트폴리오 스타트업들에 투자 완료된 금액의 합계.
    * **투자 잔액 (가용 재원)**: `[총 결성액 - 누적 투자 집행액]`을 실시간 자동 계산하여 수치 및 프로그레스 바(Progress Bar) 그래프로 출력합니다.
* **출자자(LP) 지분율 배분 차트 (Donut/Pie Chart)**
  * **기능**: 선택된 펀드의 출자 참여 기관 및 법인/개인 조합원들의 지분 배분 현황을 시각화합니다.
  * **주요 항목**: LP명, 출자 약정액, 지분율(%), 실 납입액.

### 1.1.1 펀드(조합) 리스트뷰 — 데이터 테이블
STARTUP 풀 목록(`StartupPoolTab`)의 골격을 재사용합니다: 공용 `DataTable` + 검색 `Input` + `MultiSelectFilter` + 컨테이너 4계층.

* **컬럼 순서(확정)**: 펀드명 · **재원**(모태/비모태) · 성격(개인/벤처) · **유형**(AC/VC/PE) · **상태** · 존속기간 · 약정총액 · 실출자금액 · 집행액 · 잔액(약정−집행) · 대표펀드매니저 · 운용인력 · **담당자**(우측)
  * 컬럼·필터 헤더는 "구분"을 뗀 **재원 / 성격 / 유형**으로 표기한다. 유형(AC/VC/PE)은 사이드바 탭 축이자 목록 컬럼으로도 노출한다(전체 뷰에서 조합별 전략을 바로 식별).
  * **좌측 표준**(DataTable 소유): 체크박스 · No.
  * **우측 표준**(DataTable 소유): 담당자 · 수정일. **담당자 컬럼의 값은 등록자(`created_by`)**이며, 사용자 결정으로 헤더 라벨만 `담당자`로 표기합니다(`authorLabel`). *DataTable은 이 재라벨을 권장하지 않음(작성자≠담당자 원칙)을 인지한 의도적 결정.*
* **상태 라벨**(저장값은 코드, 표시만 한글): `RAISING` 결성 중 · `OPERATING` 운용 중 · `LIQUIDATING` 청산 중 · `CLOSED` 청산 완료. 배지 톤은 운용 중=활성, 청산 완료=회색.
* **탭 상속 구조**: 대시보드의 펀드 테이블 하나를 **AC/VC/PE 탭이 상속**하여 `유형구분`으로 프리필터만 달리 겁니다(`StartupPoolTab(category=…)` 패턴). `mine`=내 담당(등록자/대표펀드매니저=나), `dashboard`=전체.
* **구현 상태**:
  * **완료(전체 컬럼)**: 펀드명·재원구분·성격구분·상태·결성일·존속기간·약정총액·실출자금액·집행액·잔액·대표펀드매니저·운용인력·담당자 + **체크박스·페이지네이션**. 검색(펀드명·매니저), 재원/성격/상태 필터, **AC/VC/PE 탭 프리필터**(유형구분), 내 펀드 스코프.
  * **마이그레이션**: 구분 4종 enum `20260724100000`, 기간·실출자 컬럼 + `fund_managers`(운용/관리 인력 junction) `20260724110000`. 등록 폼에 구분·기간·실출자 입력 반영.
  * **후속(남은 1건)**: **운용인력 배정 UI**(사람 추가/대표 지정) — `fund_managers` 테이블·조회·컬럼은 완료, 사람을 배정하는 입력(유저 피커)만 남음. 그 전까지 운용인력 컬럼은 '-'.

### 1.1.2 펀드 상세 (2:1 카드 섹션 + 서브 탭)
NETWORKS/STARTUP/AC 상세와 동일한 `grid lg:grid-cols-3`(좌 `col-span-2` : 우 `col-span-1`) 카드 레이아웃.
* **상단 바**: 뒤로가기 + **삭제**(`DetailDeleteButton`, 확인창형 소프트 삭제) · **편집**. 편집은 페이지형 폼(`FundForm`)으로 인라인 전환한다.
* **좌측(2/3)**: 펀드 개요 `CardShell`(펀드명 + 재원·성격·유형·상태 배지 + KPI 4타일〔약정·실출자·집행·잔액〕 + 정보행〔결성일·존속기간·운용기간·출자 방식·**대표펀드매니저·운용인력·등록자**·수정일〕) **고정**, 그 아래 **서브 탭바**(`Tabs`: 출자자(LP)〔도넛+표〕 / 포트폴리오 / 캐피탈 콜) — AC `ProgramOverviewTab` 구조.
* **우측(1/3)**: AC 상세와 **완전 동일한 공용 패널** — 전자결재 · 관련회의록 · 자료관리 · 코멘트 · 변동이력(타임라인은 펀드에 모듈 개념이 없어 제외). 다형 키 `'fund'`를 마이그레이션 `20260724120000`으로 지원: `app.entity_key_workspace`에 `fund` 분기 + `entity_feedback`·`entity_contributions` 정책에 `can_access_fund` arm + `funds` 변동이력 트리거 + 회의록 링크 `'fund'` 허용. 자료관리는 attachments 정책이 워크스페이스 무관이라 그대로 재사용.
* **생성·편집**: 모달이 아니라 **페이지형 폼**(`FundForm`, STARTUP `StartupDetailForm` 패턴). 생성은 `/fund/new`(`FundCreatePage`), 편집은 상세에서 인라인 전환. 필드: 펀드명·결성액·실출자 + 재원/성격/유형/출자/상태 + 결성일·존속기간·운용기간.

### 1.2 포트폴리오 투자 집행 기록
* **피투자 스타트업 매핑 표 (`Table`)**
  * **기능**: 해당 펀드로부터 투자를 유치한 스타트업 목록 및 세부 조건을 기록하고 관리합니다.
  * **주요 항목**: 피투자사명(NETWORKS 연동), 투자 집행일, 집행 금액, 취득 지분율(%), 투자 라운드(Seed, Pre-A, Series A 등), 투자방식(`RCPS`/보통주/`CPS`).
* **자사 투자 매핑 규칙**:
  * 투자실 담당자가 이 표에 투자 내역을 등록하면, 매핑된 스타트업의 상세 프로필 내 '투자 유치 내역' 섹션에 **[자사 투자 연동]** 형태의 강조 테두리 액센트와 함께 실시간 반영됩니다. 이를 통해 전사 직원이 어떤 스타트업이 자사 포트폴리오사인지 STARTUP과 OFFICE의 관련 화면에서 한눈에 구분할 수 있게 합니다.

### 1.3 캐피탈 콜 (Capital Call) 관리 스케줄러
투자 약정액을 차수별로 분할 납입받는 캐피탈 콜 업무를 트래킹합니다.

* **차수별 캐피탈 콜 타임라인**
  * **기능**: 조합원들에게 납입 요청한 내역을 차수별로 등록 및 관리합니다.
  * **주요 항목**: 캐피탈 콜 회차(예: 1차, 2차), 총 요청 금액, 납입 요청 기한(Due Date), 전체 납입 완료율(%).
* **조합원별 납입 현황 및 미납 LP 알림 콘솔**
  * **기능**: 해당 차수에 매핑된 각 LP의 납입 상태(`미납`/`완료`) 및 납입 완료 일자를 기록합니다.
  * **미납 관리**: '미납' 상태의 LP가 존재하고 납입 기한이 도래했을 경우, 해당 행 옆에 [알림톡 발송] 또는 [이메일 촉구] 버튼이 활성화되어, 단 한 번의 클릭으로 납입 촉구 안내 메시지를 즉시 발송할 수 있도록 기능을 기획합니다.

> **딜메이커 전권 모델(사용자 확정):** 딜메이커 교체는 **투자 승격 시점**에만 일어나며 기존 `promote_to_invested()` RPC로 처리합니다("기존 리드 해제 → 새 리드/지원 upsert" 원자 실행). 예: 염재민이 발굴·보육(공동관리)했더라도, 투자 승격 시 딜메이커가 리드로 지정되어 전권을 가져갑니다. 발굴·보육·기타 단계는 현행대로 공동관리이며, 별도 담당자 원장을 두지 않습니다. **이 흐름은 신규 마이그레이션 없이 기존 DB 자산만으로 성립합니다.**

## 4. 투자기업 사후관리 보고 워크플로우 (요구사항 1 부속)

> [!IMPORTANT]
> 사후관리보고서의 본문(투자포인트·사업진행현황·재무·Exit방안 등)은 **투자기업이 GUEST 앱에서 직접 입력·첨부**하고, 투자실 담당자는 이를 검토·보완요청하며, 확정된 데이터가 영업보고서(§3.2 8항)로 자동 편입되는 흐름을 목표로 합니다.

### 4.1 GUEST 앱 기업 직접 입력
* **입력 창구**: `apps/guest` 외부 앱에서 투자기업 담당자가 초대 링크/계정으로 로그인해 자사 사후관리 항목을 직접 입력·파일 첨부합니다.
* **입력 항목**: 재무현황(연도별 재무제표), 사업진행현황, 신규사업, 정부지원사업, 전년 대비 성과(고용/매출), 후속투자, Exit방안, 첨부파일.
* **보안 경계**: 투자기업은 자사 레코드에만 접근(GUEST RLS). GP 내부 데이터(밸류·회수·관리보수 등)는 절대 노출하지 않습니다.

### 4.2 보완요청 루프
* 담당자가 제출본을 검토 후 항목별 **보완요청(코멘트+상태 `REVISION_REQUESTED`)**을 걸면 GUEST 측에 재입력 요청이 전달됩니다.
* 상태 흐름: `DRAFT` → `SUBMITTED` → (`REVISION_REQUESTED` ⇄ `SUBMITTED`) → `APPROVED`.

### 4.3 재무제표 자동 표 정리
* 기업이 입력/첨부한 재무 수치를 **연도별 정형 표(매출/영업이익/당기순익/자산·부채·자본·자본금)로 자동 정리**하여 `portfolio_financials`에 적재하고, 세부현황 화면·영업보고서에 그대로 렌더링합니다.

### 4.4 자동 메일 발송 스케줄
* **시기 맞춤 발송**: 보고 주기(온기/반기/월간, 재원구분별)에 맞춰 **투자기업 담당자에게 사후관리 입력 요청 메일을 자동 발송**합니다.
* **구현**: Supabase Edge Function + 스케줄러(cron)로 대상 산출→발송, 미제출/보완 대상 리마인드 재발송. (발송 채널 연동은 Phase 14 채널 인프라와 정렬.)

## 5. 빌드 순서 및 스키마 확장 계획

### 5.1 프론트 우선(Frontend-first) 순서 — 확정
보고서 자동생성은 후순위로 놓고, **화면을 먼저 만들고 마이그레이션은 뒤에** 붙입니다. 이것이 가능한 이유는 핵심 뼈대가 **이미 DB에 존재**하기 때문입니다.

| 이미 존재(그대로 사용) | 후속 마이그레이션(뒤에) |
| :--- | :--- |
| `funds` / `fund_lps` / `capital_calls` / `investments` / `portfolio_financials` | §2.1~2.2 신규 컬럼(재원·성격·출자구분, 보수율, 실출자 등) |
| `startups` 마스터(대표자·설립일·소재지·업종) | §2.4 신규 원장 9종(회수·배분·재무·관리보수·실사·고용·Valuation·감사인) |
| `startup_managers`(리드/지원) + `promote_to_invested` RPC — **딜메이커 전권 교체** | GUEST 사후관리 입력 경로 RLS·자동 메일 Edge Function |

* **1단계(프론트)**: 위 "이미 존재" 자산 위에 FUND 화면(펀드·LP·투자기업 마스터, 딜메이커 지정/전권 교체)을 구현. 딜메이커 교체는 기존 `promote_to_invested` RPC를 그대로 호출하므로 **보안이 서버에서 이미 강제**됩니다(프론트 우선이어도 권한 공백 없음).
* **2단계(마이그레이션)**: §2 신규 컬럼·원장을 추가하고, 1단계 화면에서 임시로 비워 둔 필드를 실제 컬럼에 연결.
* **3단계(보고서)**: 영업보고서·운용보고 산출과 GUEST 자동 메일.

> [!NOTE]
> 프론트 우선 원칙의 예외: **딜메이커 전권 판정처럼 보안이 걸린 로직은 UI로 미루지 않습니다.** 다행히 해당 RPC/RLS는 이미 구현되어 있어 1단계에서 바로 붙입니다. "UI에서 숨기는 것은 보안이 아니다" 원칙(개발 수칙) 위배 없음.

### 5.2 스키마 확장 원칙
* **확장 원칙**: `20260705170000_fund_schema.sql`의 6개 테이블을 살리고, §2 필드를 컬럼 추가 및 신규 원장(§2.4)으로 보강합니다. 기존 화면(`features/fund/*`)은 점진 교체합니다.
* **보안 게이트**: 모든 신규 테이블은 FUND 스코프 RLS(`can_read/write_workspace('fund') + can_access_fund`)를 상속하고, GUEST 입력 경로는 별도 정책으로 격리합니다. 신규 테이블/RPC/`SECURITY DEFINER`/Storage 정책은 `11_migration_security_gate.md` 게이트를 통과해야 완료로 봅니다.
* **집계 트리거**: `drawn_amount` 자동 집계(`app.sync_fund_drawn_amount`) 패턴을 회수·배분·실출자 집계에도 동일 적용(`SECURITY INVOKER`).

## 6. 외부 워크스페이스 상호 참조 관계

* **NETWORKS 참조**: 투자 대상이 되는 스타트업 및 출자자(LP)가 속한 법인/기관의 사업자 정보는 `NETWORKS` 디렉토리의 공통 마스터 정보를 참조하여 유기적으로 엮입니다.

* **GUEST 연동**: 투자기업 사후관리 입력(§4)은 `apps/guest` 외부 앱을 창구로 하며, 확정 데이터가 FUND 세부현황·영업보고서로 편입됩니다.
