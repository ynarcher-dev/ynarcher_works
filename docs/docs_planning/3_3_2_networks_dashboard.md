# [3-3-2] NETWORKS 대시보드 및 최근 업로드 현황 기획서

> [!NOTE]
> 본 문서는 [3_3_workspace_networks.md](./3_3_workspace_networks.md)의 마스터 원장을 대상으로, NETWORKS 워크스페이스의 **대시보드 탭(`?tab=dashboard`)** 화면 요건을 규정하는 세부 요건서입니다. 대시보드는 마스터를 **소유·집계**만 하며 조작(등록/수정)은 각 디렉토리 탭·상세페이지가 담당합니다. NETWORKS가 직접 보유한 데이터만을 원천으로 하며, 타 워크스페이스(AC·매칭 등)가 생성하는 활동·만족도 집계에는 **의존하지 않습니다**.

---

## 1. 목적

* 운영자가 우리가 보유한 인적·기업 네트워크 자산의 **규모·구성·성장 추이·데이터 건강도**를 한 화면에서 파악하도록 합니다.
* 마스터 원장(SSOT)의 가치인 **데이터 품질**(누락·중복·미분류)을 지표화하여, 클렌징·분류 등 다음 조치의 우선순위를 제시합니다.
* 대량 업로드·개별 등록으로 최근 유입된 데이터를 **시간순으로 검수**할 수 있는 최근 업로드 현황을 제공합니다.

---

## 2. 이 문서가 다루는 범위

* NETWORKS 대시보드 탭의 정보 구조 및 위젯 구성.
* 네트워크 규모·구성(구분/분야/스타트업 풀) 인포그래픽.
* 신규 등록 성장 추이 및 유입 경로 시각화.
* 데이터 건강도 지표(미분류 대기·필드 완성도·중복 의심·담당자 배정).
* 최근 업로드된 DB 레코드 통합 리스트 및 검수 동선.
* **제외**: 개별 레코드의 등록/수정 폼(디렉토리 탭·상세페이지 소관), 활동량·멘토링 만족도 등 타 워크스페이스 집계 지표(연동 완료 후 별도 확장).

---

## 3. 핵심 사용자

* **네트워크 운영자 (internal_operator)**: 데이터 유입 상태와 미분류·중복을 점검하고 품질을 관리합니다.
* **네트워크 책임자 (internal_admin)**: 전사 네트워크 자산의 규모·구성·성장 추이를 경영·보고 관점에서 검토합니다.
* **일반 임직원 (internal_staff)**: 우리가 어느 분야·구분의 네트워크를 얼마나 보유했는지 조회합니다.

## 4. 정보 구조 (Information Architecture)

```
[NETWORKS 대시보드 (?tab=dashboard)]

 ├── 2. 네트워크 구성 인포그래픽

 │    └── 스타트업 풀 구성 (스택 막대: 구분 × 관리현황)
 ├── 3. 성장 & 유입 추이

 │    ├── 중복 의심 카운트 (→ 병합 콘솔)
 │    └── 담당자 배정 현황 (투자기업 리드/지원)
 └── 5. 최근 업로드 데이터 리스트 (8종 통합 · 최신순 · 검수 동선)
```

## 6. 주요 기능

### 6.1 규모 요약 KPI 위젯

* **총 보유 네트워크**: 8종(BAN·EXP·전문가·투자사·기업·기관·대학·외주/거래·기타) 활성 레코드 총합. 미분류·소프트삭제는 제외합니다.
* **이번 달 신규**: 당월 `created_at` 기준 신규 등록 수와 전월 대비 증감률(%)을 병기합니다.
* **미분류 대기**: `others`(미분류 데이터베이스)의 활성 건수. 클릭 시 미분류 디렉토리 탭으로 이동합니다.
* **매칭 가능 인력**: `profile.match_available`이 '가능'(값 없음=기본 가능)인 인물 수와 전체 대비 비율. (NETWORKS 자체 입력값이므로 본 대시보드 범위에 포함)

### 6.2 네트워크 구성 인포그래픽

* **구분별 분포 (도넛)**: `ENTITY_ORDER` 8종 + 미분류의 건수 비중을 도넛으로 표현합니다. 계열 색은 [7_chart_visualization_rules.md](../docs_design/7_chart_visualization_rules.md)의 `chart.1~8`을 순차 적용하고, 9번째(미분류)는 `chart.8`(Gray)로 고정합니다.
* **분야(expertise) 분포 (Treemap)**: `expertise`(ADMIN `field_tags` 다중선택) 태그별 보유 인원을 상위 N개 Treemap 또는 가로 막대로 노출해 "우리가 강한 도메인"을 드러냅니다.
* **스타트업 풀 구성 (스택 막대)**: 스타트업 마스터의 `management_status`(발굴/보육/투자/기타, [3_3_1](./3_3_1_startup_pool_classification.md) 코드 4종)를 축으로, 각 막대를 `pool_status`(관리현황 라이프사이클)로 세분한 누적 막대로 표현합니다. 관리현황이 숨김인 구분(발굴·보육)은 단색 처리합니다.

### 6.3 성장 & 유입 추이

* **월별 신규 등록 추이 (라인)**: 최근 12개월 `created_at` 집계를 구분별 계열 라인으로 표현합니다. 계열이 많을 경우 상위 5종 + 기타 합산으로 축약합니다.
* **유입 경로 비중**: `profile.source`가 `bulk_upload`인 레코드와 그 외(개별 등록)의 비율을 도넛으로 표현하여, 데이터 축적이 대량 이관 위주인지 상시 등록 위주인지 판별합니다.

### 6.4 데이터 건강도 & 관리 책임

* **미분류 대기 큐**: `others` 건수와 최장 방치 기간(가장 오래된 `created_at` 경과일)을 함께 표기합니다. 클릭 시 미분류 디렉토리로 진입해 인라인 구분 지정으로 즉시 이관합니다.
* **필드 완성도 게이지**: 이메일·연락처·분야·구분 등 핵심 필드의 채움률(%)을 막대 게이지로 노출하여 클렌징 우선순위를 제시합니다. (개인정보 값 자체는 노출하지 않고 비율만 집계)
* **중복 의심 카운트**: 이메일·전화 정규화 기준 중복 후보 그룹 수를 표기하고, 클릭 시 병합 콘솔(`MergeConsole`)로 연결합니다.
* **담당자 배정 현황**: 투자기업(`invested`) 중 리드 담당자 미배정 건수를 경고 배지로 노출합니다(비투자 구분은 공동관리이므로 집계 대상 제외).

### 6.5 최근 업로드 데이터 리스트 (Recent Uploads Feed)

* **기능**: 8종 마스터에 최근 유입된 레코드를 **엔티티 구분 없이 하나의 시간순 리스트로 통합**하여, 방금 올라온 데이터를 즉시 검수·확인할 수 있게 합니다.
* **목록 컬럼**: 이름/기업명, 구분(엔티티 라벨 배지), 소속, 등록자(`created_by`), 유입 경로(`profile.source`: 대량 업로드/개별 등록), 등록일시(`created_at`), 상세 바로가기.
* **정렬·필터**: 기본 `created_at` 내림차순. 구분·유입 경로·등록자로 필터하고, 기간(오늘/이번 주/이번 달) 프리셋을 제공합니다.
* **배치별 보기**: 대량 업로드 배치(파일 해시·업로드 이력) 단위로 그룹핑하여, 특정 업로드 묶음이 몇 건·언제·누가 올렸는지 검수할 수 있는 보조 뷰를 제공합니다.
* **개인정보 마스킹**: 리스트 셀의 이메일·연락처는 목록 마스킹 규칙을 그대로 적용하며, 상세는 상세페이지 진입 후 확인합니다.

## 7. 데이터 모델 (대시보드 뷰 전용)

대시보드는 반복 집계로 인한 부하를 피하기 위해 집계 RPC 또는 `materialized view`가 반환하는 요약 구조를 소비합니다.

```typescript
interface NetworksDashboardSummary {
  totals: {
    active_total: number;        // 8종 활성 합계(미분류 제외)
    new_this_month: number;      // 당월 신규
    mom_delta_percent: number;   // 전월 대비 증감률
    unclassified: number;        // others 활성 건수
    match_available: number;     // profile.match_available='가능' 인원
  };
  by_category: { key: string; label: string; count: number }[];         // 구분별 분포
  by_expertise: { tag: string; count: number }[];                       // 분야 상위 N
  startup_pool: { status: string; by_pool_status: Record<string, number> }[]; // 풀 구성
  monthly_new: { month: string; by_category: Record<string, number> }[]; // 12개월 추이
  source_mix: { bulk_upload: number; manual: number };                  // 유입 경로
  health: {
    field_completion: Record<'email' | 'phone' | 'expertise' | 'category', number>; // 채움률(%)
    duplicate_groups: number;    // 중복 의심 그룹 수
    unassigned_invested: number; // 담당자 미배정 투자기업 수
    oldest_unclassified_days: number; // 미분류 최장 방치일
  };
}

interface RecentUploadItem {
  entity: string;              // 대상 엔티티 키
  entity_label: string;        // 구분 라벨 배지
  id: string;                  // 레코드 UUID
  name: string;                // 이름/기업명
  affiliation: string | null;  // 소속
  created_by_name: string;     // 등록자
  source: 'bulk_upload' | 'manual';
  batch_id: string | null;     // 대량 업로드 배치 식별자
  created_at: string;
}
```

## 8. 상태 모델

* 대시보드 자체는 상태 전이를 소유하지 않는 **읽기 전용 집계 화면**입니다.
* 참조하는 라이프사이클 상태는 각 원장의 정의를 따릅니다 — 스타트업 구분/관리현황은 [3_3_1](./3_3_1_startup_pool_classification.md), 미분류→분류 이관은 디렉토리 탭의 인라인 구분 지정 흐름을 재사용합니다.

---

## 9. 권한/RLS

* **임직원 (internal_admin, internal_operator, internal_staff)**: 집계 요약 및 최근 업로드 리스트를 조회(`SELECT`)할 수 있습니다. 리스트의 개인정보 필드는 목록 마스킹이 적용됩니다.
* **외부 게스트 (guest_*)**: 대시보드 및 집계 RPC 접근이 원천 차단됩니다.
* 집계는 개별 레코드 RLS를 우회하지 않으며, 호출자 권한 범위 내 레코드만 카운트되도록 `SECURITY DEFINER` 함수 내부에서 `current_app_role()` 검증을 선행합니다.

---

## 10. API/RPC/서버 액션

* **함수명**: `fn_get_networks_dashboard_summary()`
  * **설명**: 규모·구성·추이·건강도 요약(`NetworksDashboardSummary`)을 일괄 반환합니다.
  * **보안 검증**: 호출자가 내부 임직원인지 `current_app_role()`로 검증하고, 게스트 호출 시 예외 또는 빈 구조를 반환합니다.
* **함수명**: `fn_get_networks_recent_uploads(p_limit int, p_source text, p_entity text, p_from timestamptz)`
  * **설명**: 8종 마스터를 UNION 하여 최신순 업로드 리스트(`RecentUploadItem[]`)를 페이지네이션과 함께 반환합니다. 개인정보 컬럼은 서버에서 마스킹하여 내려보냅니다.
* 두 함수 모두 신규 `SECURITY DEFINER` 함수이므로 [11_migration_security_gate.md](../docs_dev/11_migration_security_gate.md)의 보안 게이트를 통과해야 완료로 봅니다.

## 11. GUEST 연동

* 대시보드 및 최근 업로드 리스트는 게스트에게 노출되지 않습니다.
* 게스트 유입 데이터(신청서→NETWORKS 정규화)가 마스터로 승격되면 최근 업로드 리스트에 `manual`/`bulk_upload`와 구분되는 유입 경로로 표기할 수 있으나, 이는 후속 확장 항목으로 둡니다.

---

## 12. OFFICE/ADMIN/타 워크스페이스 연동

* **ADMIN 태그 원장**: 구분·분야 분포는 ADMIN이 관리하는 `category_tags`·`field_tags`·`industry_tags`를 라벨 원천으로 사용합니다. 레거시(태그 미등록) 값은 '기타'로 합산합니다.
* **OFFICE 조회**: 본 대시보드는 마스터를 소유한 NETWORKS 전용이며, OFFICE는 별도의 전사 대시보드·검색 뷰를 유지합니다(중복 구현하지 않음).
* **활동·만족도 지표**: `_activity`·`_satisfaction`은 현재 임시값(집계 미연동)이므로 본 대시보드에서 **표시하지 않습니다**. AC·매칭 실집계가 연동된 뒤 별도 확장 절로 편입합니다.

## 13. 예외/오류/운영 리스크

* **집계 지연**: 대규모 8종 UNION 집계가 느려질 수 있으므로, 요약 지표는 `materialized view`(주기 갱신) 또는 5분 캐시로 즉시 반환하고 리스트만 실시간 조회합니다.
* **데이터 편중 왜곡**: 대량 업로드 직후 특정 구분·분야가 급증해 추이가 왜곡될 수 있으므로, 월별 추이는 유입 경로(대량/개별)를 분리 조회할 수 있게 합니다.
* **빈 상태(Empty State)**: 집계 결과가 0건인 초기 상태에서도 위젯 골격과 안내 문구를 유지하여 로딩 실패와 구분되게 합니다.

## 15. 테스트 기준

1. **집계 정확도 테스트**: 특정 구분에 레코드 1건을 신규 등록한 뒤 규모 KPI '이번 달 신규'와 구분별 분포 카운트가 각 1 증가하는지 확인합니다.
2. **미분류 연동 테스트**: `others`에 레코드를 생성하면 미분류 대기 큐 카운트가 증가하고, 클릭 시 미분류 디렉토리로 진입하는지 확인합니다.
3. **최근 업로드 정렬 테스트**: 서로 다른 엔티티에 시차를 두고 등록한 레코드가 통합 리스트에서 `created_at` 내림차순으로 정확히 정렬되는지 확인합니다.
4. **유입 경로 구분 테스트**: 대량 업로드(`source='bulk_upload'`)와 개별 등록 레코드가 유입 경로 비중과 리스트 배지에서 각각 올바르게 분류되는지 확인합니다.
5. **RLS 테스트**: 게스트 토큰으로 `fn_get_networks_dashboard_summary`·`fn_get_networks_recent_uploads` 호출 시 차단(예외 또는 빈 결과)되는지 확인합니다.
6. **마스킹 테스트**: 최근 업로드 리스트의 이메일·연락처가 목록 마스킹 규칙대로 내려오는지(원본 미노출) 확인합니다.
