# [5] 공통 컴포넌트 규격 및 표준 디자인 시스템 정의서

본 문서는 와이앤아처 통합 Works 플랫폼의 시각적 일관성과 설계 효율성을 보장하기 위해, 컴포넌트 제작 시 적용해야 하는 **표준 글로벌 토큰(Radius, Shadow, Density)**과 **공통 UI 컴포넌트의 물리적 규격 명세**를 정의합니다.

> [!IMPORTANT]
> 본 문서의 수치는 코드가 원천입니다. 값 자체는 `tailwind-preset.mjs`가, "어느 맥락에 어느 값을 쓰는가"라는 매핑은 `packages/ui/src/densityScale.ts`가 소유합니다. 규격을 조정할 때는 그 두 파일을 고치고 본 문서를 뒤따라 갱신합니다. 문서에 수치를 다시 적어두고 코드와 어긋나게 두지 않습니다.

---

## 1. 글로벌 디자인 토큰 (Global Design Tokens)

### 1.1 모서리 둥글기 (Border Radius)

라운드는 "아주 약간"만 적용해 카드·버튼·테이블 모두 각진 인상을 유지합니다. 업무용 원장 화면이 주된 무대이므로, 둥근 모서리로 부드러움을 주기보다 정보 밀도와 격자감을 우선합니다.

| 토큰 | 값 | 사용처 |
| :--- | :--- | :--- |
| `radius-sm` | `2px` | 배지(`Badge`), 태그 칩, 페이지네이션 버튼 |
| `radius-md` | `4px` | 버튼, 입력 필드, 셀렉트, 드롭다운 팝오버 패널, 테이블 래퍼 |
| `radius-lg` | `6px` | 카드(`Card`·`CardShell`·`PanelCard`), 모달 다이얼로그 |
| `rounded-full` | — | 토글 스위치, 아바타, 배지 내 상태 점(dot) |

### 1.2 그림자 효과 (Box Shadow)

* **`shadow-soft`**: `0 2px 8px rgba(0, 0, 0, 0.04)` — 카드, 입력 필드, 테이블 래퍼 등 가벼운 구분감.
* **`shadow-popover`**: `0 8px 20px rgba(0, 0, 0, 0.06)` — 드롭다운 메뉴, 입력 포커스 상태.
* **`shadow-dialog`**: `0 12px 32px rgba(0, 0, 0, 0.08)` — 모달 팝업.

### 1.3 밀도 맥락 (Density Context)

> [!IMPORTANT]
> **크기를 가르는 축은 중요도가 아니라 놓이는 자리입니다.** 같은 '수정' 버튼이라도 페이지 툴바에 있으면 40px, 카드 안이면 32px, 표 셀 안이면 24px입니다. 버튼에 `size` prop을 두어 호출부가 매번 크기를 고르는 방식은 쓰지 않습니다 — 같은 자리의 컨트롤이 화면마다 달라지는 원인이 그것이었습니다.

맥락은 3종이며, `DensityProvider`가 React 컨텍스트로 하위 트리에 전달합니다.

| 맥락 | 무대 | 컨트롤 | 아이콘 버튼 | 배지 | 글자 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `page` | 페이지 툴바 · 상세 헤더 · 독립 폼 | `40px` | `36px` | `22px` | `14px` |
| `card` | 카드섹션 내부 | `32px` | `28px` | `20px` | `13px` |
| `table` | 데이터 테이블 셀 내부 | `24px` | `24px` | `18px` | `12px` |

* **맥락은 자동으로 상속됩니다**: `Card`·`CardShell`·`PanelCard`는 `card`를, `DataTable`은 `table`을 하위에 내려줍니다. 그래서 카드 안에 놓인 `<Button>`은 아무것도 지정하지 않아도 32px이 됩니다.
* **수동 지정은 예외 경로입니다**: 카드 안이지만 페이지급 강조가 필요한 경우에만 `density="page"`처럼 명시합니다. 기본값과 같은 값을 다시 적지 않습니다.
* **오버레이는 맥락을 되돌립니다**: 포털은 DOM만 `body`로 옮길 뿐 React 컨텍스트는 부모를 그대로 따릅니다. 따라서 `Modal`·`Drawer`는 `page`로, `Dropdown`·`Toast`는 `card`로 맥락을 재설정합니다. 이 처리가 없으면 표 셀에서 연 모달의 입력 필드가 24px로 렌더됩니다.
* **아이콘 버튼이 한 단계 작은 이유**: 라벨이 없어 같은 높이면 시각적으로 더 커 보이기 때문입니다. 컨트롤과의 단차는 4~8px을 유지합니다.

### 1.4 타이포그래피 스케일

| 토큰 | 크기 | 용도 |
| :--- | :--- | :--- |
| `title-lg` / `title-md` / `title-sm` | `30` / `24` / `20px` | 페이지 제목 계열 |
| `body-lg` | `16px` | 강조 본문 |
| `body` | `14px` | 본문 표준, `page` 맥락 컨트롤 라벨 |
| `body-sm` | `13px` | `card` 맥락 컨트롤 라벨 |
| `caption` | `12px` | 보조 설명, `table` 맥락 컨트롤 라벨 |
| `tag-page` / `tag-card` / `tag-table` | `12` / `11` / `10px` | 배지·태그 전용 |

* **컨트롤 라벨을 본문보다 한 단계 누르는 이유**: 조작 요소가 읽을거리보다 앞으로 튀어나오지 않게 하기 위함입니다.
* **`tag-*`는 배지 전용입니다**: `line-height`를 `1`로 고정해 부모의 행간을 상속하지 않게 했습니다. 상속하면 같은 배지가 화면마다 다른 높이로 보입니다.
* **임의값 금지**: `text-[13px]`처럼 클래스에 수치를 직접 적지 않습니다. ESLint가 차단합니다.

### 1.5 아이콘 스타일 규칙

* **표준 패밀리**: `Lucide Icons`. `stroke-width: 2px`를 유지해 시각적 무게를 통일합니다.
* **글리프 크기**: 맥락을 따릅니다 — `page` `18px` / `card` `16px` / `table` `14px`.

## 2. 기본 조작 컴포넌트 명세 (Control Components)

### 2.1 버튼 (Button & IconButton)

* **크기**: `size` prop이 없습니다. 높이·글자·좌우 여백·간격을 모두 밀도 맥락(§1.3)에서 가져옵니다.
* **위계는 색이 만듭니다**: `primary`(브랜드 채움) → `secondary`(회색 채움) → `outline`(테두리) → `ghost`(배경 없음) 순으로 무게가 낮아집니다.
* **파괴적 액션 2종**:
  * **`danger`**: 적색 채움. 단독으로 서는 결정적 삭제에 사용합니다.
  * **`outline-danger`**: 평소에는 중립이고 호버에서만 위험색이 드러납니다. 표의 '비활성화'처럼 **행마다 반복되는** 파괴적 액션용으로, 빨간 버튼이 목록에 깔리는 것을 피합니다.
* **IconButton**: 정사각 아이콘 버튼(`outline`·`ghost`·`selected` 3종 + `danger` 플래그). `label`이 필수 prop이므로 `aria-label`이 구조적으로 강제됩니다.
* **직접 만들지 않습니다**: `<button>`에 `h-* rounded border px-*`를 조합해 버튼 모양을 새로 만들지 않습니다. 상자를 가진 아이콘 액션은 전부 `IconButton`을 씁니다. ESLint가 컨트롤 요소의 임의 높이 지정을 차단합니다.

### 2.2 입력 필드 (Input / Select / TextArea)

* **크기**: 버튼과 같은 밀도 맥락을 따르므로, 한 줄에 나란히 놓인 입력과 버튼의 높이는 자동으로 일치합니다.
* **4상태 공통 외형**: 기본 · 호버(`border-gray-400`) · 포커스(`border-brand/50` + `shadow-popover`) · 비활성(`bg-gray-50`). 오류 상태는 `aria-invalid`와 함께 `border-danger-700`로 덮어씁니다.
* **아이콘 슬롯**: 좌측(검색 입력) 또는 우측(셀렉트 화살표)에 놓이며, 아이콘 위치와 반대편 여백이 짝을 이루도록 `controlIconPad`가 함께 관리합니다.
* **TextArea**: 높이는 `rows`가 정하므로 글자·여백만 맥락에 반응합니다. 최소 높이 `120px`를 보장하고 세로 리사이즈만 허용합니다(`resize-y`) — 가로 리사이즈는 레이아웃을 깨뜨립니다.
* **웹 접근성**: 플레이스홀더를 항목 명칭 대용으로 쓰지 않고, 좌측 또는 상단에 영구 텍스트 라벨을 노출합니다.

### 2.3 선택 컨트롤 (Checkbox / Radio)

* **박스 크기**: `page` `20px` / `card` `16px` / `table` `14px`.
* **둥글기**: 체크박스는 `rounded`, 라디오는 완전 원형.
* **세로 정렬**: `align-middle`을 기본 적용합니다. 인라인 요소라 그냥 두면 글자 베이스라인에 걸려 위로 떠 보입니다. 표 셀에서는 셀 자체를 flex 중앙 정렬로 두어 확실히 고정합니다.
* **라벨 연동**: `Radio`에 `label`을 넘기면 `<label>`로 감싸 클릭 영역을 라벨까지 넓힙니다. 비활성 input은 브라우저가 마우스 이벤트를 삼키므로, `title` 안내는 라벨 래퍼가 받습니다.

### 2.4 토글 스위치 (Switch)

트랙과 썸이 함께 움직여야 하므로 별도 규격표를 둡니다. 썸 이동 거리는 `트랙 너비 − 썸 크기 − 좌우 여백`과 일치시킵니다.

| 맥락 | 트랙 | 썸 | 이동 거리 |
| :--- | :--- | :--- | :--- |
| `page` | `44 × 24px` | `20px` | `22px` |
| `card` | `36 × 20px` | `16px` | `18px` |
| `table` | `28 × 16px` | `12px` | `14px` |

### 2.5 아바타 (Avatar)

* **크기**: `page` `40px` / `card` `32px` / `table` `24px`. 이니셜 글자도 함께 옮겨 원 안에서 같은 비율로 보이게 합니다.
* **둥글기**: 완전 원형.
* **대체 텍스트**: 이미지가 없으면 이름 첫 글자(영문은 대문자)를 노출합니다.

## 3. 구조 및 데이터 컴포넌트 명세 (Structure & Data Components)

### 3.1 데이터 테이블 (Data Table)

* **테이블 래퍼**: `border-gray-300` 테두리의 라운드 박스(`radius-md`) 안에 렌더링하며, 가로 스크롤 영역(`overflow-x-auto`)을 확보합니다. 카드·필드·드롭다운 등 모든 컨테이너 외곽은 동일하게 `gray-300`을 쓰고, 내부 divider만 한 단계 연한 `gray-200`을 씁니다.
* **격자**: 머리글·본문 행 모두 `row` 토큰(`36px`)으로 고정합니다. 셀 안 컨트롤이 24px이므로 위아래 6px씩 남습니다. 셀 좌우 여백은 `px-2.5`로, 셀 안 버튼의 좌우 여백과 같은 값이라 열이 시각적으로 정렬됩니다.
* **세로선**: `divide-x`로 셀 사이에만 긋고 표 바깥 가장자리에는 남기지 않습니다. 굵기는 가로선과 동일(머리글 `gray-300` / 본문 `gray-200`)하게 맞춰 한쪽 방향만 도드라지지 않게 합니다.
* **정렬**: 머리글·본문 모두 **기본 가운데**입니다. 머리글은 값 정렬과 무관하게 항상 가운데여서 머리글 줄이 하나의 띠로 읽힙니다. 이름·소속·이메일처럼 길이가 들쭉날쭉해 좌측 시작선이 필요한 열만 `align: 'left'`로 되돌립니다. 기본값과 같은 `align: 'center'`를 컬럼마다 적어두지 않습니다.
* **수치 정렬 정합성**: 수량 컬럼은 자릿수가 세로로 맞도록 `tabular-nums`를 적용합니다.
* **인터랙션**: 클릭 가능한 행은 `cursor-pointer hover:bg-gray-25`, 선택된 행은 `bg-brand/5`로 강조합니다.
* **표준 메타 컬럼(필수)**: 레코드성 테이블은 다음을 반드시 포함하며 `DataTable`이 기본 탑재합니다.
  * **좌측 `No.`**: 최상단 행이 가장 큰 번호가 되도록 내림차순으로 매깁니다.
  * **우측 `등록자`**: 최초 등록자(`created_by`). 담당자(관리 주체)는 별개 축이므로 이 컬럼을 재라벨하지 않고 도메인 컬럼으로 따로 둡니다.
  * **우측 `수정일`**: `updated_at`을 `YYYY-MM-DD`로 절삭합니다.
  * **우측 `관리`**: 물리 삭제 금지 원칙에 따라 **소프트 삭제**를 수행합니다. 삭제가 아니라 `deleted_at`을 채우므로 명칭을 '비활성화'로 통일하고 `outline-danger` 버튼을 씁니다. 이미 비활성인 행은 버튼 대신 `비활성` 텍스트로 표기하고 행 전체를 `opacity-50`으로 흐리게 처리합니다.
  * **opt-out 대상**: 감사·다운로드 로그(불변 기록), 매칭 매트릭스(N:N 격자), 랭킹(집계 뷰)처럼 레코드 성격이 아닌 표는 `numbered={false} standardColumns={false}`로 제외합니다.

### 3.2 리스트 페이지네이션 (Pagination)

* **버튼 크기**: 아이콘 버튼 격자를 그대로 씁니다(`page` 기준 `36px`). 숫자 버튼은 같은 높이를 유지하되 세 자리 이상에서 넓어지도록 최소 너비로 잡습니다.
* **활성 페이지**: 테두리 없이 브랜드 배경(`bg-brand`) + `text-white`. 비활성 번호는 `text-gray-700` + `hover:bg-gray-100`.
* **퀵 점프**: 이전(◀) 왼쪽에 맨 앞으로(⏮), 다음(▶) 오른쪽에 맨 뒤로(⏭)를 필수 탑재합니다.
* **직접 이동창**: 가장 우측에 '페이지 이동' 라벨과 숫자 입력창을 두어 번호를 직접 입력하고 엔터로 이동할 수 있게 합니다.

### 3.3 탭 메뉴 (Tabs)

* **높이**: 같은 맥락의 컨트롤과 동일합니다(`page` `40px` / `card` `32px` / `table` `24px`). 가변 세로 패딩(`py-*`)을 쓰지 않습니다 — 글꼴에 따라 높이가 흔들립니다.
* **스타일(Underline Tab)**: 카드형이 아닌 하단 라인형을 기본으로 하며, 탭 아래를 가로지르는 `1px` 가이드 라인(`border-b border-gray-200`)을 배치합니다.
* **활성 탭**: 두께 `500`, 브랜드 색, 하단 `2px` 강조 테두리. **비활성 탭**: `text-gray-600`, 호버 시 `text-gray-800`.
* **건수 칩**: 배지와 같은 규격(§3.4)을 공유합니다. 나란히 놓였을 때 어긋나지 않게 하기 위함입니다.

### 3.4 배지 및 태그 (Badge / TagChip)

* **크기 변형(`size` prop)을 두지 않습니다**: 크기는 맥락이 정합니다.

| 맥락 | 높이 | 글자 | 좌우 여백 | 점(dot) |
| :--- | :--- | :--- | :--- | :--- |
| `page` | `22px` | `12px` | `8px` | `6px` |
| `card` | `20px` | `11px` | `6px` | `4px` |
| `table` | `18px` | `10px` | `6px` | `4px` |

* **둥글기 `radius-sm`**, 테두리 `1px`(상태색), 그림자 없음, 글자 두께 `500`.
* **크기로 위계를 만들지 않습니다**: 배지가 전달하는 위계는 크기가 아니라 **색(tone)** 입니다. 같은 종류의 정보를 크기로 나누면 "큰 태그 = 더 중요한 상태"라는 잘못된 신호를 줍니다. 강조가 필요하면 톤을 바꾸고 크기는 건드리지 않습니다.
* **세로 크기는 높이로 못 박습니다**: 세로 여백(`py-*`)이 아니라 고정 높이 + `leading-none`으로 정의합니다. 세로 패딩만 주면 배지 높이가 부모의 `line-height` 상속에 끌려다닙니다.
* **색은 상태에만 씁니다**: 값이 하나뿐인 분류(예: '구분')는 배지로 감싸지 않고 텍스트로 둡니다. 분류와 상태가 같은 색 무게로 읽히면 표를 훑을 때 어느 쪽이 지금 확인할 행인지 구분되지 않습니다.
* **직접 만들지 않습니다**: `<span>`에 `rounded + px + py`를 조합해 태그 모양을 새로 만들지 말고 `Badge`(표시용) 또는 `TagChip`(선택 토글용)을 씁니다. 둘은 같은 규격표를 공유해 나란히 놓여도 어긋나지 않습니다.
* **상태 매핑 4종**:
  * **완료/성공(Success)**: 글씨 `#059669` | 배경 `#ECFDF5` | 보더 `#D1FAE5`
  * **대기/주의(Warning)**: 글씨 `#D97706` | 배경 `#FFFBEB` | 보더 `#FEF3C7`
  * **진행/정보(Info)**: 글씨 `#0064FF` | 배경 `#F0F5FF` | 보더 `#E0EBFF`
  * **취소/중단(Danger)**: 글씨 `#EF4444` | 배경 `#FFF5F5` | 보더 `#FFE2E2`
  * 위 4종 외에 분류·중립 표기는 `neutral`(회색)을 씁니다.

### 3.5 다이얼로그 모달 (Dialog / Modal)

* **너비**: Tailwind 기본 `max-w-sm/lg/2xl`은 타이포그래피 척도라 다이얼로그 폭으로 쓰면 의미가 어긋납니다. 대화 단계별 전용 토큰을 씁니다.

| 토큰 | 폭 | 용도 |
| :--- | :--- | :--- |
| `modal-sm` | `400px` | 확인·경고 등 한 문장짜리 대화 |
| `modal-md` | `600px` | 단일 폼(기본값) |
| `modal-lg` | `800px` | 2열 폼 · 목록 선택 |
| `modal-xl` | `1000px` | 표를 품는 대화 |
| `modal-2xl` | `1200px` | 전체 화면에 준하는 편집기 |

* **모서리** `radius-lg`, **그림자** `shadow-dialog`.
* **밀도**: 어디서 열든 `page` 맥락으로 재설정합니다(§1.3).
* **구조**: 머리글·바닥글은 고정하고 본문만 스크롤합니다(최대 높이 = 뷰포트 − 여백).
* **푸터 버튼 정렬**: 항상 우측 정렬하며, 보조 버튼(취소·닫기)을 좌측에, 주요 액션(저장·등록)을 가장 우측에 배치합니다.

### 3.6 오버레이·피드백 (Dropdown / Drawer / Toast / Banner / EmptyState)

* **Dropdown**: 팝오버 패널은 `radius-md` · `border-gray-300` · `shadow-popover`. 항목은 카드 규격 높이(32px)로 고정하며 가변 세로 패딩을 쓰지 않습니다.
* **Drawer**: 모달과 같은 이유로 밀도를 `page`로 되돌립니다.
* **Toast**: 앱 어디서 호출하든 같은 크기여야 하므로 밀도를 `card`로 고정합니다. 자동 소멸 `4초`, 우측 하단 배치.
* **Banner**: 화면·폼 상단의 인라인 알림. 배지와 같은 상태 4색을 씁니다.
* **EmptyState**: 제목 + 설명 + 액션 슬롯의 중앙 정렬 블록.

### 3.7 규격 확인 페이지

`/styleguide`(WORKS 내부 전용, 메뉴 미노출)에서 같은 코드가 세 맥락에 놓였을 때의 결과를 나란히 확인할 수 있습니다. 각 컴포넌트의 실제 렌더 높이를 px로 함께 표시하므로, 규격 조정 여부는 문서가 아니라 이 화면을 보고 판단합니다.

> [!NOTE]
> `tailwind-preset.mjs`는 Vite의 감시 대상 밖입니다. 토큰 값을 고친 뒤에는 개발 서버를 **재시작**해야 화면에 반영됩니다. 컴포넌트 소스(`.tsx`) 수정은 HMR로 즉시 반영됩니다.

## 4. AC Program 운영 특화 컴포넌트 후보

AC 워크스페이스의 기능 모델은 `matching` 레퍼런스의 운영 감각을 가져오되, 화면 구성은 Works 공통 디자인 시스템을 따른다. 따라서 아래 컴포넌트는 별도 UI 복제가 아니라 기존 AppShell, PageHeader, Tabs, DataTable, Drawer, Modal, StatusBadge, SplitView 패턴을 조합하는 방식으로 정의한다.

### 4.1 ProgramCard / ModuleCard
* **ProgramCard**는 Program 목록과 대시보드에서 사용하며, Program명, 상태, 기간, 담당자, 활성 모듈 수, 미해결 운영 이슈 수를 표시한다.
* **ModuleCard**는 Program 상세의 모듈 보드에서 사용하며, `module_type`, `module_status`, `participation_mode`, 진행률, 마지막 업데이트 시각, 바로가기 액션을 포함한다.

### 4.2 EvaluationSplitView / FileSubmissionPanel
* **EvaluationSplitView**는 서면평가, 대면평가, 데모데이 평가에서 사용한다.
  * 좌측에는 신청서, PDF, 발표자료, 기업 요약 정보를 표시한다.
  * 우측에는 평가표, 점수 입력, 의견 입력, 임시저장/제출 액션을 배치한다.
* **FileSubmissionPanel**은 모집 신청, 발표자료 제출, 세션 자료 제출에서 사용하며, 파일 상태, 제출 마감, 검토 상태, 다운로드 권한을 함께 표시한다.

### 4.3 ScheduleSlotGrid / AssignmentMatrix
* **ScheduleSlotGrid**는 멘토링 회차, 비즈니스 매칭 슬롯, 대면평가 발표 시간표에서 사용한다.
* **AssignmentMatrix**는 심사위원-기업, 멘토-멘티, 전문가-상담 기업의 N:N 또는 1:1 배정 상태를 한 화면에서 확인하도록 구성한다.
* 두 컴포넌트 모두 시간 충돌, 배정 중복, 권한 범위 위반을 StatusBadge와 Inline Alert로 표시한다.

### 4.4 CounselingLogEditor / KpiSummaryPanel / ExportColumnPicker
* **CounselingLogEditor**는 멘토링 및 비즈니스 매칭 상담일지 작성에 사용하며, 요약, 이슈, 조언, 액션아이템, 후속 일정 필드를 기본으로 가진다.
* **KpiSummaryPanel**은 Program KPI와 모듈별 운영 지표를 카드형 요약으로 표시한다.
* **ExportColumnPicker**는 Excel 다운로드 전 컬럼 그룹, 마스킹 정책, 다운로드 사유를 선택하게 하며, Export 권한이 없는 사용자에게는 비활성 상태로 노출한다.

### 4.5 GuestTaskCard / MobileActionBar
* **GuestTaskCard**는 GUEST 포털의 오늘 할 일, 평가 제출, 예약 확인, 출석 체크, 상담일지 작성, 만족도 제출을 표시한다.
* **MobileActionBar**는 모바일 GUEST 화면 하단에 고정되는 주요 액션 영역이며, 한 화면에 하나의 핵심 제출/확정 액션만 강조한다.
